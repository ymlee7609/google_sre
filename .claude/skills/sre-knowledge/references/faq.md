# SRE FAQ 캐시

자주 묻는 SRE 질문에 대한 사전 답변입니다. 스킬이 질의 시 먼저 이 파일에서 Grep 검색하여 매칭되면 즉시 답변합니다.

---

## Q: SLO와 에러 버짓이란 무엇인가?

**SLO(Service Level Objective)**는 서비스가 달성해야 할 신뢰성 목표를 정량적으로 정의한 것입니다. SLI(Service Level Indicator)를 기반으로 측정하며, "99.9% 요청이 200ms 이내 응답"과 같은 형태입니다.

**에러 버짓(Error Budget)**은 100%에서 SLO를 뺀 나머지로, 허용 가능한 불안정성의 양입니다. 예를 들어 99.9% SLO의 에러 버짓은 0.1%입니다. 에러 버짓이 남아있으면 새 기능 출시나 변경을 진행하고, 소진되면 안정성 작업에 집중합니다.

- SLO는 완벽함(100%)을 목표로 하지 않습니다. 적절한 신뢰성 수준을 선택하는 것이 핵심입니다
- SLA(Service Level Agreement)는 SLO에 비즈니스적 결과(패널티)가 결합된 계약입니다
- SLO는 반복적으로 개선해야 합니다. 처음부터 완벽할 필요 없습니다

출처: [SRE Book ch03, ch04], [Workbook ch02, ch03]

## Q: 4 Golden Signals(4대 골든 시그널)이란?

모니터링의 핵심 4대 지표입니다:

1. **Latency(지연시간)**: 요청 처리 시간. 성공 요청과 실패 요청의 지연시간을 구분해야 합니다
2. **Traffic(트래픽)**: 시스템에 대한 수요 측정. HTTP 요청/초, 트랜잭션/초 등
3. **Errors(오류율)**: 실패한 요청의 비율. 명시적(5xx) + 암묵적(200이지만 잘못된 콘텐츠) 포함
4. **Saturation(포화도)**: 시스템 리소스 사용률. CPU, 메모리, I/O, 네트워크 등의 포화 정도

이 4가지만 잘 모니터링해도 대부분의 문제를 감지할 수 있습니다. "심플하게 시작하라"가 원칙입니다.

출처: [SRE Book ch06], [Workbook ch04]

## Q: 토일(Toil)이란 무엇이고 어떻게 제거하나?

**토일(Toil)**은 프로덕션 서비스 운영과 관련된 수작업으로, 다음 특성을 가집니다:
- **수동적(Manual)**: 사람이 직접 수행
- **반복적(Repetitive)**: 같은 작업을 반복
- **자동화 가능(Automatable)**: 기계가 대신할 수 있음
- **전술적(Tactical)**: 장기 가치 없이 단기 대응
- **서비스 성장에 비례(O(n))**: 서비스 규모에 따라 증가

Google SRE는 토일을 50% 미만으로 유지하는 것을 목표로 합니다. 나머지 50%는 엔지니어링 작업에 투자합니다.

제거 전략: 자동화, 셀프서비스 도구, 프로세스 개선, 불필요한 작업 제거

출처: [SRE Book ch05], [Workbook ch06]

## Q: 인시던트 발생 시 대응 절차를 알려줘

Google의 인시던트 관리 프로세스:

1. **감지(Detection)**: 모니터링/알림으로 문제 인지
2. **대응(Response)**: 온콜 엔지니어가 초기 대응
3. **인시던트 선언**: 영향이 크면 공식 인시던트 선언
4. **역할 분담**:
   - **인시던트 커맨더(IC)**: 전체 조율, 의사결정
   - **운영 리드(Ops Lead)**: 기술적 대응 수행
   - **커뮤니케이션 리드**: 이해관계자 소통
5. **커뮤니케이션**: 라이브 인시던트 문서 작성, 상황실(War Room) 운영
6. **완화(Mitigation)**: 근본 원인 해결보다 서비스 복구 우선
7. **포스트모템**: 비난 없는(blameless) 사후 분석

핵심 원칙: "먼저 서비스를 복구하고, 나중에 원인을 분석하라"

출처: [SRE Book ch13, ch14], [Workbook ch09], [BSRS ch17]

## Q: 포스트모템(Postmortem)은 어떻게 작성하나?

포스트모템은 인시던트 후 학습을 위한 문서입니다:

**핵심 원칙: Blameless(비난 없는 문화)**
- 개인이 아닌 시스템/프로세스의 실패에 초점
- "누가 했나"가 아니라 "왜 시스템이 이를 허용했나"를 질문

**포스트모템 구성**:
1. 인시던트 요약 및 영향도
2. 타임라인 (감지 → 대응 → 완화 → 해결)
3. 근본 원인 분석 (5 Whys 등)
4. 교훈 (잘한 점, 부족한 점)
5. 액션 아이템 (재발 방지 조치, 담당자, 기한)

**트리거 기준**: SLO 위반, 데이터 손실, 사용자 영향, 수동 개입 필요 등

출처: [SRE Book ch15], [Workbook ch10], [SRE Book appendix_d]

## Q: 온콜(On-Call) 운영 모범 사례는?

효과적인 온콜 운영 원칙:

- **균형 잡힌 온콜**: 엔지니어당 최대 25% 시간만 온콜, 나머지는 프로젝트 작업
- **페이지 빈도**: 하루 최대 2건의 인시던트. 이상이면 시스템 개선 필요
- **심리적 안전**: 에스컬레이션을 장려, 혼자 판단하지 않아도 되는 문화
- **충분한 보상**: 온콜 수당, 대체 휴무 보장
- **명확한 에스컬레이션 경로**: 1차 → 2차 → 매니저 → 다른 팀
- **플레이북/런북**: 일반적인 알림에 대한 대응 절차 문서화
- **운영 과부하 방지**: 온콜 부하가 과도하면 팀 확장 또는 시스템 개선

출처: [SRE Book ch11, ch28, ch29], [Workbook ch08]

## Q: 카오스 엔지니어링이란 무엇이고 어떻게 시작하나?

**카오스 엔지니어링**은 시스템의 약점을 발견하기 위해 의도적으로 장애를 주입하는 방법입니다.

Google의 **DiRT(Disaster Recovery Testing)**:
- 정기적으로 대규모 장애 시나리오를 시뮬레이션
- 네트워크 분리, 데이터센터 장애, 의존성 장애 등 테스트
- 복구 절차와 런북의 유효성 검증

시작 방법:
1. 가장 중요한 서비스의 단일 장애점(SPOF) 식별
2. 정상 상태 가설 수립 (SLO 기반)
3. 작은 범위에서 장애 주입 시작 (한 서버, 한 프로세스)
4. 결과 관찰 및 학습
5. 점진적으로 범위 확대

핵심: 프로덕션에서 테스트하되, 안전장치를 마련하라

출처: [SRE Book ch17], [BSRS ch16]

## Q: 로드 밸런싱과 과부하 처리 전략은?

**로드 밸런싱 계층**:
1. DNS 기반 (Anycast, 지리적 분산)
2. VIP/프록시 기반 (L4/L7 로드 밸런서)
3. 백엔드 레벨 (서브셋 선택, 가중 라운드로빈)

**과부하 처리 전략**:
- **부하 차단(Load Shedding)**: 용량 초과 시 일부 요청 거부
- **우아한 성능 저하(Graceful Degradation)**: 기능 축소로 핵심 서비스 유지
- **요청 중요도(Criticality)**: CRITICAL > SHEDDABLE+ > SHEDDABLE 순으로 처리
- **클라이언트 측 스로틀링**: 적응형 재시도로 서버 부하 감소
- **데드라인 전파**: 요청 체인에서 타임아웃 전파로 무의미한 작업 방지

출처: [SRE Book ch19-22], [Workbook ch11, ch17]

## Q: 연쇄 장애(Cascading Failure)를 어떻게 방지하나?

연쇄 장애의 주요 원인:
- 서버 과부하 → 다른 서버로 부하 전이
- 리소스 고갈 (CPU, 메모리, 스레드, 파일 디스크립터)
- 건강하지 않은 재시도 (Retry Storm)

방지 전략:
1. **부하 차단과 우아한 성능 저하**: 과부하 시 핵심 기능만 유지
2. **재시도 제한**: 지수 백오프 + 지터(jitter) 적용, 재시도 예산 설정
3. **데드라인 전파**: 이미 만료된 요청은 처리하지 않음
4. **서킷 브레이커**: 실패율 임계치 초과 시 요청 차단
5. **용량 계획**: 피크 부하 + N+2 여유분 확보
6. **장애 격리(Bulkhead)**: 장애 영역을 격리하여 전파 차단
7. **부하 테스트**: 한계점을 사전에 파악

출처: [SRE Book ch21, ch22], [Workbook ch17], [BSRS ch08]

## Q: SRE와 DevOps의 차이점은?

SRE는 DevOps 철학의 구체적 구현입니다:

| 측면 | DevOps | SRE |
|------|--------|-----|
| 정의 | 문화/운동 | 구체적 직무/방법론 |
| 초점 | 개발-운영 협업 | 서비스 신뢰성 |
| 측정 | 다양 | SLO/SLI/에러 버짓 |
| 토일 | 자동화 강조 | 50% 상한 규칙 |
| 채용 | 다양한 배경 | 소프트웨어 엔지니어링 역량 필수 |

핵심 관계: "SRE는 DevOps의 class SRE implements DevOps"

출처: [SRE Book ch01], [Workbook ch01]

## Q: 시스템 단순성(Simplicity)은 왜 중요한가?

복잡성은 신뢰성의 적입니다:

- 복잡한 시스템은 장애 원인 파악이 어렵고 복구가 느림
- 모든 코드는 부채(liability)이며, 최소한의 API가 최선
- "지루한 기술(Boring Technology)"을 선호: 검증된 기술이 안정적
- 필수 복잡성과 우발적 복잡성을 구분하라
- 정기적으로 사용하지 않는 기능/코드 제거

"소프트웨어 단순성은 신뢰성의 전제조건이다"

출처: [SRE Book ch09], [Workbook ch07], [BSRS ch06]

## Q: 제로 트러스트(Zero Trust) 아키텍처란?

전통적 경계 보안(perimeter security) 대신 모든 접근을 검증하는 보안 모델:

- **기본 원칙**: "네트워크 위치를 신뢰하지 않는다"
- **모든 요청 인증/인가**: 내부 네트워크라도 검증
- **최소 권한(Least Privilege)**: 필요한 최소한의 권한만 부여
- **접근 분류**: 읽기/쓰기, 데이터 민감도 기반
- **지속적 검증**: 세션 중에도 지속적으로 신뢰 평가
- **안전한 프록시**: 모든 관리 작업은 프록시를 통해 수행

구현 단계: 자산 식별 → 접근 정책 정의 → 정책 엔진 구축 → 점진적 적용

출처: [BSRS ch03, ch05]

## Q: 서킷 브레이커와 복원력 패턴은?

시스템 복원력을 위한 핵심 패턴:

- **서킷 브레이커(Circuit Breaker)**: 실패율 임계치 초과 시 요청 차단, 일정 시간 후 재시도
- **격벽(Bulkhead)**: 장애 영역을 격리하여 전체 시스템 보호
- **부하 차단(Load Shedding)**: 과부하 시 우선순위 낮은 요청 거부
- **스로틀링(Throttling)**: 요청 속도 제한
- **심층 방어(Defense in Depth)**: 여러 계층의 보호 장치
- **폭발 반경 제한(Blast Radius)**: 변경/장애의 영향 범위 최소화
- **장애 도메인 분리**: 독립적인 장애 영역 설계

출처: [BSRS ch08], [SRE Book ch22]

## Q: 릴리스 엔지니어링과 카나리 배포란?

**릴리스 엔지니어링**: 코드를 안전하게 프로덕션에 배포하는 프로세스
- 자동화된 빌드/배포 파이프라인
- 재현 가능한(Hermetic) 빌드
- 빠른 롤백 능력

**카나리 배포(Canary Release)**:
1. 전체 트래픽의 일부(1-5%)를 새 버전으로 라우팅
2. 핵심 메트릭(SLO) 모니터링
3. 문제 없으면 점진적 확대, 문제 발견 시 즉시 롤백
4. 카나리 기간: 최소 1시간, 중요 서비스는 24시간+
5. 비교 대상: 카나리 vs 기준(baseline) 인스턴스

출처: [SRE Book ch08], [Workbook ch16], [BSRS ch14]

## Q: 데이터 무결성(Data Integrity) 전략은?

데이터 손실은 가장 심각한 장애입니다:

**핵심 원칙**:
- 데이터 손실 방지가 데이터 접근성보다 중요
- 1차 방어: 소프트 삭제(soft deletion) + 지연 삭제
- 2차 방어: 백업 및 복구 (정기적 검증 필수!)
- 3차 방어: 조기 감지 (데이터 검증 파이프라인)

**Google의 접근**:
- 백업 vs 아카이브를 구분하라
- 복구 테스트를 정기적으로 실행하라
- 복구 SLO를 정의하라 (RPO/RTO)
- "백업은 있지만 복구를 테스트한 적 없다"는 백업이 없는 것과 같다

출처: [SRE Book ch26], [Workbook ch13]

## Q: 재해 복구(Disaster Recovery) 계획은 어떻게 수립하나?

**재해 유형**: 자연재해, 인프라 장애, 보안 사고, 소프트웨어 버그, 인적 오류

**계획 수립 단계**:
1. 리스크 분석: 위협 식별 및 영향도/확률 매트릭스
2. 복구 목표 정의: RPO(Recovery Point Objective), RTO(Recovery Time Objective)
3. 대응 전략 수립: 런북/플레이북 작성
4. 인시던트 대응 팀 구성: 역할, 연락처, 에스컬레이션 경로
5. 사전 준비(Prestaging): 복구에 필요한 리소스 사전 배치
6. 정기적 테스트: DiRT 훈련으로 계획 유효성 검증

핵심: "계획을 테스트하지 않으면 계획이 없는 것과 같다"

출처: [BSRS ch09, ch16], [SRE Book ch17]

## Q: SRE 팀을 어떻게 구성하고 운영하나?

**팀 구성**:
- 소프트웨어 엔지니어링 역량을 갖춘 인력 채용
- 50% 엔지니어링 / 50% 운영 시간 배분 원칙
- 6-8명이 이상적인 온콜 로테이션 크기

**참여 모델(Engagement Model)**:
1. **얼리 인게이지먼트**: 설계 단계부터 SRE 참여
2. **PRR(Production Readiness Review)**: 서비스 인수 전 점검
3. **점진적 이전**: 운영 책임을 서서히 이전
4. **SRE 해제**: 서비스가 SRE 기준을 충족하지 못하면 운영 반환

**문화**:
- 비난 없는 문화 (Blameless Culture)
- 데이터 기반 의사결정
- 토일 감소에 대한 조직적 지원

출처: [SRE Book ch30-32], [Workbook ch18-21]

## Q: 트러블슈팅(Troubleshooting) 체계적 방법론은?

체계적 문제 해결 프로세스:

1. **문제 보고(Problem Report)**: 증상 정확히 기록, 예상 동작 vs 실제 동작
2. **분류(Triage)**: 영향도/긴급도 평가, 즉각 완화 조치 (원인 분석보다 우선)
3. **조사(Examine)**: 로그, 메트릭, 트레이스 확인
4. **진단(Diagnose)**: 가설 수립 → 검증 → 반복 (이분법 접근)
5. **테스트 & 치료(Test/Treat)**: 수정 적용, 검증, 모니터링

핵심 원칙:
- "관찰 → 가설 → 검증"의 과학적 방법론
- 동시에 여러 가설을 검증하지 말 것 (한 번에 하나씩)
- "이 시스템이 동작해야 하는 방식"을 먼저 이해하라

출처: [SRE Book ch12]

## Q: 위협 모델링(Threat Modeling)은 어떻게 하나?

위협 모델링은 시스템의 보안 위험을 체계적으로 식별하는 프로세스입니다:

**공격자 프로파일**:
- 취미 해커, 범죄 조직, 내부자, 국가 수준 위협
- 각 프로파일별 동기, 역량, 자원이 다름

**방법론**:
1. 자산 식별 (무엇을 보호할 것인가)
2. 공격 표면 분석 (어디에서 공격 가능한가)
3. 위협 식별 (누가 왜 공격하는가)
4. 리스크 평가 (확률 x 영향도)
5. 완화 전략 수립

**내부자 위협(Insider Risk)**:
- 가장 과소평가되는 위협 중 하나
- 최소 권한, 접근 로깅, 이중 승인으로 완화
- 기술적 통제와 조직적 통제 병행

출처: [BSRS ch02]

## Q: 보안 코딩(Secure Coding) 핵심 원칙은?

안전한 코드 작성을 위한 원칙:

- **프레임워크 활용**: 보안 기능이 내장된 프레임워크 사용 (직접 구현 금지)
- **입력 검증**: 모든 외부 입력은 악의적이라고 가정
- **SQL Injection 방지**: 파라미터화된 쿼리(Prepared Statement) 사용
- **XSS 방지**: 자동 이스케이핑, Content Security Policy 적용
- **기본값 안전(Security by Default)**: 안전한 설정이 기본값이 되도록 설계
- **단순성**: 코드가 단순할수록 보안 취약점이 줄어듦
- **코드 리뷰**: 보안 관점의 코드 리뷰 프로세스

출처: [BSRS ch12]

## Q: 설정 관리(Configuration Management) 모범 사례는?

안전하고 효율적인 설정 관리:

- **설정은 코드처럼 관리**: 버전 관리, 코드 리뷰, 테스트
- **설정 검증**: 배포 전 자동 검증 (스키마 체크, 드라이런)
- **점진적 롤아웃**: 설정 변경도 카나리 배포 적용
- **즉시 롤백 능력**: 문제 발생 시 이전 설정으로 즉시 복원
- **설정 단순화**: 불필요한 설정 옵션 제거, 합리적인 기본값
- **환경별 분리**: 개발/스테이징/프로덕션 설정 분리

출처: [Workbook ch14, ch15]

## Q: 용량 계획(Capacity Planning)은 어떻게 하나?

효과적인 용량 계획:

- **수요 예측**: 과거 트래픽 패턴 + 비즈니스 성장률 기반
- **N+2 원칙**: 최소 2개의 여유 인스턴스 확보 (1개 장애 + 1개 유지보수)
- **부하 테스트**: 실제 트래픽 패턴으로 시스템 한계 파악
- **오토스케일링**: 자동 확장/축소로 비용 효율성 확보
- **부하 차단 준비**: 용량 초과 시의 대응 전략 사전 수립

출처: [SRE Book ch22, ch27], [Workbook ch11, ch17]
