# Release Engineering 요약

## 핵심 원칙
- 릴리스 엔지니어링의 4대 원칙: 셀프서비스 모델, 높은 속도, 밀봉(Hermetic) 빌드, 정책/절차 강제
- 밀봉 빌드: 동일 리비전에서 다른 머신에서 빌드해도 동일한 결과를 보장한다
- 빈번한 릴리스는 버전 간 변경을 줄여 테스트와 트러블슈팅을 용이하게 한다
- 릴리스 프로세스는 재현 가능하고 자동화되어야 하며 "유니크 스노우플레이크"가 되어서는 안 된다
- 브랜치에서 릴리스하고 메인라인에 다시 머지하지 않으며, 버그 수정은 체리픽으로 포함한다
- 설정 관리는 릴리스 엔지니어와 SRE 간 긴밀한 협업 영역이다
- 릴리스 엔지니어링은 제품 개발 초기부터 예산에 포함해야 한다

## 주요 프랙티스 및 권고사항
- Rapid 시스템: 블루프린트 설정 파일로 빌드/테스트/배포 워크플로우를 정의한다
- Blaze(Bazel) 빌드 도구로 의존성 자동 빌드 및 빌드 식별자 추적을 수행한다
- MPM(Midas Package Manager)으로 패키지 이름/버전/서명을 관리하고 레이블로 단계를 표시한다
- 배포 리스크 프로파일에 맞춰 배포 전략을 차별화한다: 개발 환경(자동), 대규모 서비스(점진적 확장), 인프라(며칠에 걸친 롤아웃)
- 설정 배포 4가지 모델: 메인라인, 바이너리와 함께 패키징, 별도 설정 패키지, 외부 저장소(Chubby/Bigtable)
- 연속 테스트 시스템으로 모든 변경에 대해 단위 테스트를 실행하고, 릴리스 브랜치에서도 재실행한다
- Sisyphus 롤아웃 프레임워크로 복잡한 배포를 관리한다
- 모든 변경에 코드 리뷰를 의무화하고, 릴리스에 포함된 모든 변경사항 보고서를 생성한다

## 핵심 인용
> "Releases can be as painless as simply pressing a button."

> "It's cheaper to put good practices and process in place early, rather than have to retrofit your system later."
