# Load Balancing at the Frontend 요약

## 핵심 원칙
- 대규모 시스템에서 모든 것을 한 바구니에 담는 것은 재앙의 레시피이다 - 단일 장애점 의존은 허용할 수 없다
- 최적의 부하 분산은 단일 답이 없다 - 계층 수준(글로벌 vs 로컬), 기술 수준(하드웨어 vs 소프트웨어), 트래픽 특성에 따라 달라진다
- 검색 요청은 지연시간 최적화를 위해 가장 가까운 데이터센터로, 비디오 업로드는 처리량 최적화를 위해 다른 경로로 라우팅한다
- DNS 로드 밸런싱은 사용자 연결 시작 전 부하를 분산하는 가장 단순하고 효과적인 방법이지만, 그것만으로는 충분하지 않다
- 빛의 속도는 광섬유 통신의 물리적 한계로 작용하여, 데이터 이동 거리가 서비스 속도의 상한을 결정한다

## 주요 프랙티스 및 권고사항
- DNS 로드 밸런싱: 여러 A/AAAA 레코드를 반환하여 클라이언트가 IP를 선택하게 하는 가장 단순한 방법
- DNS의 3대 제약: 재귀 해석기(사용자 IP 대신 해석기 IP), 비결정적 응답 경로, 추가 캐싱 복잡성
- EDNS0 확장: 재귀 해석기가 클라이언트 서브넷 정보를 DNS 쿼리에 포함하여 사용자 관점에서 최적화된 응답 가능
- DNS 해석기 뒤 사용자 기반의 크기와 지리적 분포를 추적하여 응답의 잠재적 영향을 추정한다
- DNS TTL을 낮게 설정하여 변경 전파 속도의 하한선을 설정하지만, 모든 해석기가 TTL을 존중하지는 않는다
- DNS 응답은 512바이트 제한(RFC 1035) 내에 맞아야 하므로 포함 가능한 주소 수가 서버 수보다 적다
- VIP(Virtual IP Address) 로드 밸런싱: VIP는 특정 네트워크 인터페이스에 할당되지 않고 여러 장치에서 공유
- 연결 ID 기반 백엔드 선택: id(packet) mod N 방식은 백엔드 장애 시 거의 모든 연결이 재설정되는 문제 발생
- 일관된 해싱(Consistent Hashing): 백엔드 추가/제거 시에도 상대적으로 안정적인 매핑을 제공하여 기존 연결 교란 최소화
- Direct Server Response(DSR): 데이터 링크 계층에서 MAC 주소만 변경하여 백엔드가 직접 응답 - 대부분의 HTTP 트래픽에서 큰 절감 효과
- 패킷 캡슐화(GRE): Google의 현재 VIP 솔루션으로, 백엔드와 로드 밸런서가 같은 브로드캐스트 도메인에 있을 필요 없이 대륙 간에도 작동
- 캡슐화의 대가: 24바이트(IPv4+GRE) 오버헤드로 MTU 초과 시 프래그멘테이션 발생 가능 - 데이터센터 내 더 큰 MTU로 완화

## 핵심 인용
> "Putting all your eggs in one basket is a recipe for disaster."

> "Load balancing, especially for large systems, is anything but straightforward and static."
