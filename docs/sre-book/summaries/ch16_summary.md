# Tracking Outages 요약

## 핵심 원칙
- 시간이 지남에 따른 신뢰성 개선은 알려진 기준선에서 출발하여 진행 상황을 추적해야만 가능하다
- 포스트모템은 대규모 인시던트만 다루므로, 개별적으로 작은 영향이지만 빈번하고 광범위한 이슈는 별도의 추적이 필요하다
- 장애 추적 도구는 과거 데이터 분석을 통해 체계적, 주기적, 광범위한 문제를 식별하는 것이 가장 중요한 기능이다
- 팀 간 가시성(cross-team visibility)의 향상이 인시던트 해결을 크게 가속화한다
- 알림을 인시던트로 그룹화하여 "일일 인시던트 수"와 "일일 알림 수"를 분리 분석해야 한다
- 태깅은 단순해 보이지만 서비스의 문제 지점을 파악하는 데 가장 유용한 고유 기능이다

## 주요 프랙티스 및 권고사항
- Escalator 시스템: 중앙 집중식 알림 에스컬레이션 시스템으로, 사람이 확인하지 않으면 자동으로 다음 담당자에게 전달한다
- Outalator 시스템: 모니터링 시스템의 모든 알림을 수동적으로 수신하여 주석, 그룹화, 분석을 가능하게 한다
- 여러 큐의 알림을 시간순 통합 뷰로 제공하여 팀 간 알림 추적을 용이하게 한다
- 알림 집계(Aggregation): 단일 이벤트가 여러 알림을 트리거할 때 이를 하나의 인시던트로 그룹화한다
- 태깅(Tagging): 자유 형식 태그와 계층적 네임스페이스(cause:network:switch 등)를 사용하여 메타데이터를 추가한다
- 미리 정해진 태그 목록을 강제하지 않고 팀이 자체 기준과 선호도를 찾도록 허용한다
- 분석(Analysis): 팀/서비스 간, 시간 경과에 따른 비교를 통해 패턴과 추세를 식별한다
- 교대 인수인계 시 선택한 장애 정보를 이메일로 전달하여 최근 상태를 공유한다
- 주간 프로덕션 리뷰를 위한 보고 모드에서 중요 주석을 인라인 확장하여 하이라이트 개요를 제공한다
- 더미 에스컬레이터 설정을 통해 특권 계정 사용 로깅, 주기적 작업 기록 등 "기록 시스템"으로도 활용한다
- 인프라 구성 요소별 인시던트 원인 분석으로 안정성 향상의 잠재적 이득을 정량화한다
