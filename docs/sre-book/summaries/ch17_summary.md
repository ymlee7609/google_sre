# Testing for Reliability 요약

## 핵심 원칙
- 테스트하지 않았다면 고장났다고 가정하라
- 테스트는 변경이 발생할 때 동등성(equivalence)의 특정 영역을 입증하는 메커니즘이다
- 테스트 커버리지가 높을수록 각 변경으로 인한 불확실성이 줄어들어 더 많은 변경을 허용한다
- MTTR이 0인 테스트는 프로덕션 배포 전에 버그를 차단하여 사용자가 경험하는 MTBF를 증가시킨다
- 안정성과 민첩성은 전통적으로 긴장 관계지만, 빌드가 안정적이고 신뢰할 수 있으면 개발자가 더 빠르게 반복할 수 있다
- 모든 소프트웨어가 동등하지 않다 - 생명 또는 매출에 영향을 미치는 시스템은 훨씬 높은 테스트 품질을 요구한다
- 신뢰성 관리는 테스트 커버리지로 인한 불확실성 한계와 사용자 가시적 결함 한계를 결합하여 릴리스 주기를 조정한다

## 주요 프랙티스 및 권고사항
- 전통적 테스트 계층: 단위 테스트(가장 작고 빠름) → 통합 테스트(의존성 주입으로 모킹) → 시스템 테스트(엔드투엔드)
- 시스템 테스트 유형: 스모크 테스트(기본 동작), 성능 테스트(점진적 저하 방지), 회귀 테스트(이전 버그 재발 방지)
- 프로덕션 테스트 유형: 설정 테스트(실제 구성과 소스 코드 비교), 스트레스 테스트(한계 탐색), 카나리 테스트(점진적 배포)
- 카나리 테스트는 진정한 테스트가 아니라 구조화된 사용자 수용(structured user acceptance)이다
- 버그 오더(order): U=1은 선형적 결함, U=2는 데이터 손상, U=3은 식별자 연쇄 - 고차 버그는 릴리스 중 반드시 잡아야 한다
- 테스트 환경 구축: 버전 관리 시스템 → 지속적 빌드 시스템(코드 제출마다 빌드 및 테스트) → 결함 즉시 수정
- 모든 보고된 버그를 테스트 케이스로 문서화하여 포괄적 회귀 테스트 모음을 구축한다
- 프로덕션 프로브: 알려진 좋은 요청을 프로덕션에서 모니터링 프로브로 재생하여 환경 간 차이를 감지한다
- 설정 파일 테스트: 프로토콜 버퍼를 사용하여 스키마 사전 정의와 자동 검증으로 런타임 바운드를 보장한다
- 통계적 테스트(퍼징, Chaos Monkey, Jepsen)는 반복 가능하지 않지만, 무작위 동작 로그를 릴리스 테스트로 리팩토링한다
- Break-glass 메커니즘: 긴급 시 릴리스 테스트를 비활성화하되, 테스트는 계속 실행하여 결과를 역주석한다
- Bazel 같은 빌드 시스템으로 의존성 그래프 기반 선택적 테스트를 실행하여 비용과 속도를 최적화한다

## 핵심 인용
> "If you haven't tried it, assume it's broken."

> "Testing is one of the most profitable investments engineers can make to improve the reliability of their product."

> "The more bugs you can find with zero MTTR, the higher the Mean Time Between Failures experienced by your users."
