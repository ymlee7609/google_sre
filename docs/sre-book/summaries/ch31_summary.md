# Communication and Collaboration in SRE 요약

## 핵심 원칙
- SRE는 명령 통제 조직이 아니다 - 서비스 팀과 SRE 전체라는 최소 두 주인에게 충성하며, 서비스 관계가 매우 강하지만 보고 라인은 SRE 전체를 통한다
- 데이터가 프로덕션 주위를 흐르듯이, SRE 팀 주위로도 프로젝트, 서비스 상태, 프로덕션, 개인 상태에 대한 데이터가 신뢰성 있게 흘러야 한다
- SRE 팀이 제시하는 인터페이스는 API와 같다 - 좋은 설계가 효과적 운영에 필수적이며, 잘못된 API는 나중에 고치기 고통스럽다
- 최고의 설계와 구현은 프로덕션과 제품의 공동 관심사가 상호 존중의 분위기에서 만날 때 탄생한다
- 싱글톤 프로젝트는 특히 재능이 뛰어나거나 문제가 단순하지 않은 한 실패한다 - 의미 있는 성과를 위해서는 다수의 인원과 협업 능력이 필요하다

## 주요 프랙티스 및 권고사항
- 프로덕션 미팅: SRE 팀이 담당 서비스의 상태를 명확히 표현하는 주간 회의 - 서비스 지향적이며 개인 업무 보고가 아님
- 프로덕션 미팅의 목표: (1) 모두가 동일한 인식을 가지고 회의를 떠남 (2) 프로덕션의 지혜를 서비스에 적용하여 개선
- 프로덕션 미팅 의제: 예정된 변경, 메트릭, 장애, 페이징 이벤트, 비페이징 이벤트, 이전 액션 아이템
- 미팅 의장 순환: 모든 팀원에게 서비스의 소유감을 부여하고, 인시던트 조율 상황에서 유용한 의장 능력을 키움
- 비디오 미팅 팁: 두 사이트의 팀 크기가 다를 때 의장을 작은 쪽에 배치하면 큰 쪽이 자연스럽게 조용해짐
- Google Docs 실시간 협업: 의제를 병렬로 미리 작성하고 바텀업 아이디어를 사전에 추가하여 효율성과 포용성 향상
- 팀 구성: 다양성이 높을수록 팀 성과가 향상된다는 많은 증거가 있음 - TL, SRM, PM/TPM 역할의 유동적 또는 명확한 분리
- 크로스 사이트 협업: 물리적 거리는 훌륭한 서면 커뮤니케이션 또는 대면 출장으로 보완 - 시간이 지나면 이메일 주소에 불과해진다
- Viceroy 사례 연구: 모니터링 대시보드 프레임워크의 중복 개발 문제를 여러 팀이 힘을 합쳐 단일 솔루션으로 해결
- Viceroy 교훈: (1) 기여자가 실제로 헌신하는지 확인 (2) 분할 정복으로 통신 비용 절감 (3) 리더가 장기 비전 제공 (4) 합의된 의사결정 방식 수립
- Conway의 법칙: 크로스 사이트 프로젝트에서 소프트웨어 구조가 조직 구조에 왜곡되지 않도록 주의
- SRE 외부 협업: 설계 단계 초기에 SRE 참여가 가장 효과적 - 코드가 커밋되기 전에 시작하는 것이 이상적
- DFP-F1 마이그레이션 사례: SRE가 인프라 설계를 주도하고 제품 팀이 비즈니스 로직을 독립적으로 개발 - 초기 인터페이스 정의로 병렬 작업 가능
- 대면 미팅의 대체 불가성: 궁극적으로 대면 상호작용을 대체할 수 없으며, 팀 서밋을 중립 장소에서 개최하면 홈 어드밴티지를 방지
