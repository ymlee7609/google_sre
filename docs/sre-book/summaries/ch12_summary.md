# Effective Troubleshooting 요약

## 핵심 원칙
- 트러블슈팅은 가설-연역적 방법(hypothetico-deductive method)의 적용이다 - 관찰, 가설 수립, 테스트의 반복
- 트러블슈팅은 학습 가능하고 가르칠 수 있는 기술이며, 타고난 재능이 아니다
- 두 가지 핵심 요소: 범용 트러블슈팅 프로세스 이해 + 해당 시스템에 대한 깊은 지식
- 첫 번째 우선순위는 출혈 멈추기(서비스 복구)이다 - 근본 원인 분석은 그 다음이다
- 상관관계는 인과관계가 아니다 - 허위 상관에 속지 않아야 한다
- 부정적 결과(negative results)도 가치 있다 - 무엇이 작동하지 않는지 아는 것도 진보이다
- 시스템은 관성을 가진다 - 최근 변경사항이 문제의 출발점이 되는 경우가 많다

## 주요 프랙티스 및 권고사항
- 문제 보고서에 기대 동작, 실제 동작, 재현 방법을 포함시킨다
- 트리아지: "비행기를 먼저 조종하라" - 서비스 복구가 디버깅보다 우선한다
- 분할 정복(divide and conquer) 전략: 스택의 한쪽 끝에서 시작하여 체계적으로 각 컴포넌트 점검
- "무엇을, 어디서, 왜" 질문: 시스템이 현재 하고 있는 것을 파악하고 자원 사용처를 추적한다
- 테스트 설계 시 상호 배타적 대안을 가진 실험을 선호하고, 가능성 높은 가설부터 검증한다
- 모든 가설, 테스트, 결과를 명확히 기록한다 - 포스트모템과 미래 참조를 위해
- 관찰 가능성(observability)을 컴포넌트 설계 시점부터 내장한다 (화이트박스 메트릭, 구조화된 로그)
- 고유 요청 식별자로 RPC 체인 전체를 추적한다 (Dapper 등)
