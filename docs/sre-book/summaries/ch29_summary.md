# Dealing with Interrupts 요약

## 핵심 원칙
- 운영 부하는 페이지, 티켓, 지속적 운영 책임의 세 가지 범주로 나뉜다 - 각각 응답 시간 SLO가 분/시간/주 단위로 다르다
- 인간은 불완전한 기계이다 - 컨텍스트 스위칭이 무비용인 인터럽트 가능한 작업 단위로 취급하면 행복과 생산성이 모두 저하된다
- 인터럽트에 전념하고 있을 때, 인터럽트는 더 이상 인터럽트가 아니다 - 프로젝트가 오히려 산만함이 된다
- 시간을 양극화(polarize)하라 - 프로젝트 작업 또는 인터럽트 작업, 둘 중 하나에만 집중하라
- 20분의 중단은 두 번의 컨텍스트 스위치를 수반하며 현실적으로 수 시간의 생산적 작업 손실을 초래한다

## 주요 프랙티스 및 권고사항
- 인지적 몰입 상태(Cognitive Flow State): 명확한 목표, 즉각적 피드백, 통제감이 있을 때 달성 - 창의적 결과와 양적으로 좋은 성과를 낳는다
- 인터럽트 관리 메트릭: 인터럽트 SLO, 대기열 크기, 심각도, 빈도, 처리 가능 인원 수 - 인적 비용도 반드시 고려
- 온콜 엔지니어: 온콜 주간에는 프로젝트 작업을 완전히 포기해야 한다 - 온콜 중에 프로젝트 진행도 기대해서는 안 된다
- 티켓 관리: 티켓을 팀원에게 무작위로 배정하는 것을 중단하라 - 이는 팀 시간에 대한 극도의 비존중이다
- 전담 티켓 역할: 티켓 처리를 전담 역할로 만들어 관리 가능한 기간 동안 운영, 부하 초과 시 2인으로 확장하되 전체 팀에 분산하지 말라
- 지속적 운영 책임: 푸시 관리자(push manager) 역할을 정의하여 온콜/인터럽트 기간 동안 담당하고, 인계 프로세스를 공식화하라
- 인터럽트 상태 이분법: 인터럽트를 하거나 하지 않거나 - 온콜이 아닌 사람이 "바쁘게 보이기 위해" 티켓을 처리하면 티켓 부하 추정이 왜곡된다
- 티켓 분석: 인계 시 근본 원인을 실제로 조사하라 - 대부분의 팀이 온콜 인계는 하지만 티켓 인계는 하지 않아 같은 문제가 반복된다
- 정기적 티켓/페이지 정리: 인터럽트 클래스를 조사하여 근본 원인 식별 후, 근본 원인이 수정될 때까지 해당 인터럽트를 침묵시켜 핸들러에게 안도를 제공
- 자기 존중과 고객 존중의 균형: 정책을 도구로 활용하여 고객에게 노력의 일부를 되돌려 놓을 수 있다 - 정책은 코드만큼 강력한 도구이다
- 불필요한 지원 구성요소: 수정 자원을 얻지 못하는 취약한 도구는 페이저를 반납하거나 사용 중단을 고려하라

## 핵심 인용
> "When you're doing interrupts, your projects are a distraction."

> "A person should never be expected to be on-call and also make progress on projects."

## 관련 키워드
인터럽트 관리, 인지적 몰입 상태, 컨텍스트 스위칭, 시간 양극화, 페이지, 티켓, 운영 부하, 전담 역할, 티켓 분석, 인계 프로세스, 산만함, 정책 도구
