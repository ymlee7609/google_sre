# Accelerating SREs to On-Call and Beyond 요약

## 핵심 원칙
- 성공적인 SRE 팀은 신뢰 위에 구축된다 - 동료 온콜 담당자가 시스템을 이해하고, 비정상 동작을 진단하며, 도움을 요청할 수 있고, 압박 하에 대응할 수 있음을 신뢰해야 한다
- "불 속에 던지기(trial by fire)" 방식의 온보딩은 일부 엔지니어만 살아남게 하고 나머지를 소외시킨다 - SRE 훈련은 단순히 하는 것이 아닌 추론을 통해 이루어져야 한다
- SRE는 강한 역공학 능력, 통계적 사고력, 즉흥 대처 능력의 세 가지 속성을 갖추어야 한다
- 온콜은 새 SRE 경력의 이정표이며, 그 이후의 학습은 더 모호하고 자기 주도적이 된다
- SRE는 기계보다 빠르게 인력을 확장해야 한다 - 학습과 교육의 문화를 만드는 것이 필수적이다

## 주요 프랙티스 및 권고사항
- 추천 훈련 패턴: 구체적이고 순차적인 학습 경험 설계, 역공학 장려, 장애 분석 축하, 실제 모니터링/도구로 현실적 장애 제공
- 안티패턴: 단순 작업(알림/티켓)으로 훈련, 체크리스트/플레이북만으로 훈련, 장애를 비밀로 취급, 전문가의 지식 고립화
- 온콜 학습 체크리스트: 기술과 개념에 대한 체계적 읽기 및 이해 목록 - 전문가 연락처, 핵심 문서, 기본 지식, 탐구 질문으로 구성
- 학습 경로 설계: 쿼리 진입 → 프론트엔드 서빙 → 중간 계층 → 인프라 → 종합(디버깅, 에스컬레이션) 순서로 구성
- 파워업(Powerups): 학습 체크리스트 섹션 완료 시 점진적으로 시스템 접근 권한을 부여하는 계층적 접근 모델
- 프로젝트 작업: 사소하지만 의미 있는 소유권 부여 - 서빙 스택의 사소한 기능 변경, 모니터링 블라인드 스팟 추가, 아직 충분히 고통스럽지 않은 자동화
- 역공학 수업: "Google News 팀이 버뮤다 삼각지대로 갔다" 시나리오 - 학생들이 도움 없이 프로덕션 서비스를 처음부터 역공학
- Wheel of Misfortune(불운의 바퀴): 주간 재난 역할극 - 게임 마스터가 시나리오를 준비하고, 선택된 온콜 담당자가 대응을 연습, 모든 경험 수준의 팀원이 함께 학습
- 실제 시스템 파괴 및 복구: 실제 트래픽에서 분리된 프로덕션 인스턴스에서 합성 부하로 현실적 장애를 생성하고 복구 연습
- "검색 클러스터를 태워버리자!" 연습: 알려진 정상 구성에서 시작하여 선택된 병목 지점을 점진적으로 손상시키며 모니터링 관찰 - 분기별 수행
- 섀도 온콜: 학습 체크리스트 완료 후 업무 시간 중 수신 페이지를 복사하여 시간 압박 없이 장애 전면석 관람 - 여러 팀원과 섀도잉
- 역섀도: 경험 많은 온콜러가 신입의 기본 온콜을 배후에서 관찰하며 필요 시 지원, 검증, 힌트 제공
- 문서화를 도제 관계로: 신입 SRE에게 가장 오래된 학습 체크리스트 섹션의 개편을 맡겨 주제 전문가와 조기 연결을 촉진
- 포스트모템: 신입을 장애 포스트모템의 공저자로 포함하되, 절대로 신입에게만 작성을 떠넘기지 말라 - 포스트모템이 하찮은 업무라는 인상을 만들 수 있다
