# Data Integrity: What You Read Is What You Wrote 요약

## 핵심 원칙
- 데이터 무결성은 사용자가 생각하는 것이다 - 실제 데이터 손실이 없더라도 빈 메일함이 너무 오래 표시되면 사용자는 데이터가 손실되었다고 판단한다
- 데이터 무결성 없는 데이터 가용성은 의미가 없고, 데이터 가용성 없는 데이터 무결성도 사용자에게는 데이터가 없는 것과 같다
- 99.99% 가동시간은 연간 1시간의 다운타임만 허용하지만, 99.99%의 데이터 정확도는 2GB 중 200KB 손상을 의미하며 이는 대부분의 경우 치명적이다
- 뛰어난 데이터 무결성의 비결은 사전 감지(proactive detection)와 신속한 복구(rapid repair and recovery)이다
- 복제와 중복은 복구 가능성이 아니다 - 자동 동기화되는 복제본은 손상된 데이터도 모든 복사본에 전파한다
- "아무것이나 잘못될 수 있다"가 아니라 "모든 것이 잘못될 것이다"를 인식하는 것이 진정한 준비의 시작이다

## 주요 프랙티스 및 권고사항
- 백업이 아닌 복구에 집중하라: 아무도 백업을 원하지 않는다 - 사람들이 진정 원하는 것은 복원이다
- 백업과 아카이브의 구분: 백업은 애플리케이션에 다시 로드할 수 있지만 아카이브는 불가능하다 - 용도가 완전히 다르다
- 데이터 무결성 실패의 24가지 조합: 근본 원인(6가지) x 범위(광범위/좁은) x 속도(빅뱅/점진적)
- Google의 조사에서 가장 흔한 사용자 가시적 데이터 손실 시나리오는 소프트웨어 버그로 인한 데이터 삭제 또는 참조 무결성 손실이었다
- 심층 방어(Defense in Depth) 3계층: (1) 소프트 삭제 (2) 백업 및 복구 (3) 조기 감지(대역 외 데이터 검증)
- 제1 계층 - 소프트 삭제: 삭제된 데이터를 즉시 사용 불가로 표시하되 15~60일간 보존, 개발자 오류와 사용자 오류 모두에 대한 1차 방어선
- 지연 삭제(Lazy Deletion): 클라우드 API 수준에서 삭제 후에도 데이터를 수주간 보존하는 추가 보호 계층
- 제2 계층 - 계층화된 백업 전략: 빠른 복원용 로컬 스냅샷(시간~일) → 분산 파일시스템 백업(일~주) → 오프라인 테이프(주~월)
- 제3 계층 - 대역 외 데이터 검증(Out-of-band Validation): MapReduce/Hadoop 파이프라인으로 데이터스토어 내부 및 간 불변량을 지속적으로 검증
- Gmail 사례: 다수의 데이터 검증기가 실제 프로덕션 데이터 무결성 문제를 감지했으며, 개발자들은 24시간 이내 감지를 확신하고 주 1회 이상 코드 변경을 배포
- Google Drive 사례: 파일 내용과 폴더 목록 간의 정합성을 주기적으로 검증하여 2013년 잠재적 긴급 상황을 일상적인 작업으로 전환
- 복구 프로세스를 정기적으로 테스트하라: 테스트하지 않은 복구 프로세스는 필요할 때 실패한다 - 실제로 복구해야만 복구 가능함을 확인할 수 있다
- Gmail 2011년 사례: GTape 오프라인 백업 시스템으로 대규모 데이터 복구 - 정기 리허설과 DiRT 연습 덕분에 성공
- Google Music 2012년 사례: 리팩토링된 삭제 파이프라인의 경쟁 조건으로 60만 개 오디오 트랙 삭제, 5,000개 이상의 테이프에서 1.5PB 복구
- 1T 대 1E: 테라바이트에 적용되는 프로세스는 엑사바이트 규모에서 작동하지 않는다 - 신뢰 지점(trust points)과 병렬 분산을 활용해야 한다
- SRE의 데이터 무결성 원칙: 초심자의 마음(Beginner's Mind), 신뢰하되 검증(Trust but Verify), 희망은 전략이 아니다, 심층 방어
