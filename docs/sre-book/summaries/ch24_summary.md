# Distributed Periodic Scheduling with Cron 요약

## 핵심 원칙
- 단일 머신 cron의 장애 도메인은 하나의 머신이다 - 데이터센터 규모에서는 이 접근이 수용 불가능하다
- 이중 실행보다 실행 누락을 선호한다 - 뉴스레터 이중 발송의 복구가 한 번 누락보다 훨씬 어렵거나 불가능하다
- cron 같은 광범위한 영향을 미치는 기본 서비스는 최소한의 의존성을 가져야 한다
- Paxos 분산 합의 알고리즘으로 여러 복제본의 상태 일관성을 보장하며, 다수가 가용하면 전체 시스템이 작동한다
- 상태 일관성의 정확도가 가장 중요하다 - 많은 cron 작업(급여 처리, 이메일 뉴스레터)은 멱등이 아니다

## 주요 프랙티스 및 권고사항
- 대규모 cron의 요구사항: 프로세스를 머신에서 분리, 분산 파일시스템에 상태 지속, 핫 스페어로 빠른 페일오버
- Paxos 기반 복제: 리더만이 상태를 변경하고 cron 작업을 실행하며, Fast Paxos 리더와 cron 서비스 리더를 동일하게 사용
- 리더 역할: 스케줄된 시간에 도달하면 실행 시작을 Paxos를 통해 동기적으로 공지 → 실제 실행 → 완료를 동기적으로 공지
- Paxos 통신의 동기성 필수: 비동기 시 리더 장애 후 팔로워가 이미 실행된 작업을 재실행할 위험
- 리더 장애 시: 1분 이내 빠른 페일오버, 새 리더가 미완료 실행(open launches)을 해결
- 부분 장애 해결 2가지 조건: (1) 외부 시스템 작업이 멱등이거나, (2) 작업 상태를 명확히 조회 가능
- 작업 이름에 예정 실행 시간을 포함하여 중복 실행을 방지
- 팔로워 역할: 리더로부터 Paxos를 통해 상태 변경을 수신하고, 즉시 인수할 수 있도록 대기
- 상태 저장: Paxos 로그는 로컬 디스크에, 스냅샷은 로컬 디스크 + 분산 파일시스템에 백업
- 분산 파일시스템에 로그를 저장하지 않는 의식적 결정: 빈번한 소규모 쓰기의 성능 페널티가 크고, 3개 복제본 동시 손실 위험은 수용 가능
- 새 복제본은 이미 실행 중인 복제본에서 네트워크를 통해 상태를 가져와 빠르게 시작
- 선더링 허드(Thundering Herd) 방지: crontab에 물음표(?) 와일드카드를 도입하여 시스템이 실행 시간을 분산 - 작업 설정의 해시로 시간 범위에 균등 분배
- 데이터센터 내 배포 시 전력 분배 장치 장애가 전체 cron 서비스를 무력화하지 않도록 복제본을 다양한 위치에 배치

## 핵심 인용
> "We favor skipping launches rather than risking double launches, as much as the infrastructure allows."

> "Not performing this task synchronously could mean that the entire cron job launch happens on the leader without informing the follower replicas."
