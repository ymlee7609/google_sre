# The Evolving SRE Engagement Model 요약

## 핵심 원칙
- 소프트웨어 엔지니어링에서 버그를 일찍 발견할수록 수정 비용이 낮은 것처럼, SRE 상담이 빨리 이루어질수록 서비스가 더 좋아진다
- 최적의 접근은 개별 맞춤 시스템이 SRE의 문 앞에 "도착"하는 과정을 단축하고, SRE가 검증한 인프라 플랫폼을 제공하는 것이다
- SRE 참여 모델은 단순 PRR에서 조기 참여, 그리고 프레임워크/플랫폼으로 진화했으며, 세 모델 모두 현재 Google에서 실행 중이다
- 프레임워크는 제품 개발팀이 애플리케이션 레이어의 혁신에 집중하게 하면서 인프라의 신뢰성과 확장성을 보장한다
- 모든 서비스가 전면적 SRE 지원을 받을 필요는 없다 - 신뢰성 있고 유지보수가 적은 서비스는 개발팀에 남는 것이 긍정적 결과이다

## 주요 프랙티스 및 권고사항
- SRE 참여의 프로덕션 관심 영역: 시스템 아키텍처, 계측/모니터링, 비상 대응, 용량 계획, 변경 관리, 성능(가용성/지연/효율성)
- 대안적 지원: 전면 SRE 참여가 불가능할 때 문서화(프로덕션 가이드)와 컨설팅(LCE 팀)으로 개선 가능
- Simple PRR Model: 이미 출시된 서비스 대상 - 참여(Engagement) → 분석(Analysis) → 개선/리팩토링 → 훈련(Training) → 온보딩 → 지속적 개선
- PRR 분석 단계: SRE가 서비스를 학습하고 프로덕션 결함을 점검하는 체크리스트 기반 검토 - 도메인 전문성, 유사 시스템 경험, 프로덕션 가이드의 모범 사례에 기반
- PRR 체크리스트 예시: 업데이트의 영향 범위, 의존성 인스턴스 적절성, 네트워크 QoS, 오류 로깅, 사용자 가시적 요청 실패 모니터링
- Early Engagement Model: 설계 단계부터 SRE 참여 - 적용 대상은 기존 SRE 관리 시스템의 신규 기능, 대규모 재작성, 또는 출시 시 SRE 인수를 요청한 서비스
- 설계 단계 참여 이점: 프로덕션 인시던트를 사전에 예방하고 차선의 설계 트레이드오프에 대한 사전 인식 공유
- 빌드 단계 참여: 계측/메트릭, 운영 제어, 리소스 사용 효율에 대해 SRE가 특정 라이브러리/컴포넌트를 권장
- 출시 단계 참여: "다크 런치" - 기존 사용자 트래픽의 일부를 새 서비스에 전송하되 응답은 폐기하여 운영 인사이트 확보
- 출시 후: 안정적 시스템은 신뢰성 개선 vs 기능 추가의 우선순위 충돌을 줄여 개발 모멘텀을 유지
- Simple PRR의 한계: 2~3명 SRE로 2~3분기 소요, 긴 리드 타임, 서비스별 다른 구현으로 중복 작업, 교차 서비스 수정/개선 복제 불가
- 외부 요인: 마이크로서비스 증가로 서비스 수와 SRE 지원 요청 증가, SRE 채용 어려움, 직접 SRE 지원이 없는 다수 개발팀 지원 필요
- Frameworks 접근: SRE가 검증한 프로덕션 모범 사례를 코드로 구현하여 서비스가 설계에 의해 "프로덕션 준비" 상태가 되도록 함
- 프레임워크 모듈: 계측/메트릭, 요청 로깅, 트래픽/부하 관리 제어 - 비즈니스 로직에 집중하도록 인프라 사용을 캡슐화
- 프레임워크의 이점: (1) 운영 오버헤드 대폭 감소 (2) SRE 직원 장벽 해소로 모든 서비스에 보편적 지원 (3) 단일 SRE 1분기 온보딩 (4) 아이디어에서 SRE급 품질 배포까지 수일
- 공유 책임 모델: SRE가 플랫폼 인프라(부하 차단, 자동화, 트래픽 관리, 모니터링)를 담당하고 개발팀이 비즈니스 로직의 온콜을 담당하는 새로운 관계 모델

## 핵심 인용
> "The best production incidents are those that never happen!"

> "Frameworks free developers from having to glue together and configure individual components in an ad hoc service-specific manner."

## 관련 키워드
SRE 참여 모델, Production Readiness Review, 조기 참여, 프레임워크, 서비스 플랫폼, 공유 책임, 온보딩, 다크 런치, 마이크로서비스, 프로덕션 가이드, 체크리스트, 지속적 개선
