# Addressing Cascading Failures 요약

## 핵심 원칙
- 연쇄 장애는 양의 피드백(positive feedback)의 결과로 시간이 지남에 따라 성장하는 장애이다
- 대부분의 연쇄 장애는 서버 과부하에 직접적으로 기인하거나 그 변형이다
- 리소스 고갈은 서로를 먹여 살린다 - CPU 부족이 메모리 문제를, 메모리 문제가 GC 폭주를, GC가 다시 CPU 부족을 유발한다
- 서비스가 11,000 QPS에서 연쇄 장애가 시작되면, 9,000 QPS로 줄여도 회복되지 않는다 - 1,000 QPS까지 내려야 안정화된다
- 실패하면 지수적으로 백오프하고, 항상 약간의 지터(jitter)를 추가하라

## 주요 프랙티스 및 권고사항
- 서버 과부하 방지 우선순위: (1) 용량 한계 부하 테스트 (2) 저하된 결과 제공 (3) 과부하 시 요청 거부 (4) 상위 시스템에서 요청 거부 (5) 용량 계획
- 리소스 고갈 유형: CPU(요청 지연, 큐 포화, 스레드 기아), 메모리(GC 죽음 나선, 태스크 사망, 캐시 적중률 저하), 스레드, 파일 디스크립터
- 큐 관리: 스레드 풀 대비 작은 큐 크기(50% 이하) 유지, FIFO 대신 LIFO 또는 CoDel 알고리즘 사용
- 부하 차단(Load Shedding): 과부하 접근 시 태스크별 CPU/메모리/큐 기반 스로틀링, HTTP 503 반환
- 우아한 저하(Graceful Degradation): 검색 시 전체 코퍼스 대신 인메모리 캐시만 검색, 덜 정확한 랭킹 알고리즘 사용
- 재시도 모범 사례: 항상 무작위 지수 백오프 사용, 요청당 재시도 제한, 서버 전체 재시도 예산 유지(분당 60회 등)
- 데드라인 전파: RPC 데드라인을 스택 전체에 전파하여 이미 만료된 요청에 대한 불필요한 작업 방지
- 느린 시작과 콜드 캐시: 재시작 직후 태스크는 캐시가 비어 있어 더 많은 리소스를 소비한다 - 트래픽을 점진적으로 증가
- 항상 충분한 용량 유지: N+2 프로비저닝으로 가장 큰 클러스터 장애와 유지보수를 동시에 처리
- 연쇄 장애 테스트: 프로덕션과 유사한 환경에서 과부하 조건을 테스트하여 실제 장애 모드를 파악
- 저하 모드 연습: 정기적으로 서버 일부를 과부하 근처에서 운영하여 저하 코드 경로를 검증
- 연쇄 장애 대응 체크리스트: 부하 증가 중지 → 비정상 태스크 처리 → 서버 재시작 → 저하 모드 활성화 → 리소스 추가 → 로그 확인
