# A Collection of Best Practices for Production Services 요약

## 핵심 원칙
- 합리적으로 실패하라(Fail Sanely): 잘못된 입력 수신 시 이전 상태로 계속 운영하면서 알림을 보내라
- 모든 비긴급 롤아웃은 반드시 단계적으로 진행하되 감독하에 수행하라 - 예상치 못한 동작이 감지되면 먼저 롤백하고 나중에 진단하라
- 모니터링은 페이지, 티켓, 로깅의 세 가지 출력 유형만 가져야 한다 - 이메일 알림에 의존하는 것은 /dev/null로 파이프하는 것의 도덕적 등가물이다
- 포스트모템은 비난 없이(blameless) 프로세스와 기술에 초점을 맞추라 - 관련자가 이용 가능한 정보로 최선의 선택을 했다고 가정하라

## 주요 프랙티스 및 권고사항
- Fail Sanely: 구성 입력의 문법과 의미론 검증, 빈 데이터와 잘린 데이터 감시, 이전 버전 대비 N% 이상 축소 시 알림
- 2005년 DNS 사례: 빈 DNS 파일 수락으로 6분간 모든 Google 속성에 NXDOMAIN 응답 → 이후 google.com VIP 존재 확인 등 입력 검증 도입
- 2009년 맬웨어 표시 사례: "/"로 모든 URL이 매칭되어 전체 웹이 맬웨어로 표시 → 급격한 파일 크기 변경 확인 필요
- SLO 정의: 서버가 아닌 사용자 관점에서 가용성과 성능을 측정 → Gmail이 99.0%에서 99.9% 이상으로 개선
- 오류 예산: 1 - SLO = 예산, 예산 소진 시 긴급 보안/버그 수정 외 변경 동결, 성숙 서비스(99.99%+)는 분기별 리셋
- 용량 계획: N+2 구성으로 계획적/비계획적 동시 장애 처리, 부하 테스트로 리소스-용량 비율 검증, 출시일 부하와 정상 부하 구분
- 과부하 시: 합리적이지만 최적이 아닌 결과 제공(검색 시 인덱스 축소), 우아한 부하 차단, 타르피팅, 일관된 클라이언트 서브셋에 오류 전달
- 재시도: 모든 RPC 클라이언트에 지터를 포함한 지수적 백오프 필수 - 특히 모바일 클라이언트는 업데이트에 수주 소요
- SRE 팀: 운영 작업 50% 이하 유지, 최소 8명 온콜 팀, 교대당 최대 2건 이벤트 처리, 제품 개발자를 온콜 로테이션에 포함
- 역설: 모범 사례를 잘 구현하면 인시던트가 드물어져 대응 연습이 부족해질 수 있으므로 정기적 재난 역할극 필요

## 핵심 인용
> "Putting alerts into email and hoping that someone will read all of them and notice the important ones is the moral equivalent of piping them to /dev/null."

> "We can't fix the people, but must instead fix their environment."

## 관련 키워드
Fail Sanely, 점진적 롤아웃, SLO, 오류 예산, 모니터링 출력 유형, 비난 없는 포스트모템, N+2 용량, 과부하 처리, 지수적 백오프, 50% 운영 상한
