# Software Engineering in SRE 요약

## 핵심 원칙
- SRE 내 소프트웨어 엔지니어링은 일회성 솔루션이 아닌 본격적인 제품 개발 프로젝트이다
- SRE는 프로덕션의 복잡성에 대한 직접적 경험이 있어 적절한 도구를 개발할 수 있는 독보적 위치에 있다
- "팀 규모는 서비스 성장에 비례하여 직접 확장해서는 안 된다" - 자동화와 도구 개발이 필수적이다
- SRE 개발자는 자신의 제품의 소비자이자 개발자 역할을 동시에 수행하여 강한 소유 의식과 신뢰성을 확보한다
- 완벽을 추구하지 말고 출시 후 반복하라(Launch and iterate) - 특히 문제의 경계가 명확하지 않을 때
- 의도 기반(Intent-based) 접근은 구현이 아닌 요구사항을 명시하여 용량 계획의 유연성을 극대화한다

## 주요 프랙티스 및 권고사항
- Auxon 사례 연구: SRE가 개발한 의도 기반 용량 계획 및 리소스 할당 솔루션으로, 수백만 달러 규모의 머신 리소스 계획에 사용
- 전통적 용량 계획의 문제: 본질적으로 취약(brittle), 노동 집약적이고 부정확, 스프레드시트의 확장성 한계
- 의도(Intent)의 추상화 단계: "50코어 X 클러스터" → "지역 내 3개 클러스터" → "N+2 이중화" → "5나인 신뢰성" - 더 높은 추상화가 더 큰 유연성 제공
- 의도의 3대 전제조건: 의존성(서비스 간 관계), 성능 메트릭(수요-자원 변환), 우선순위(트레이드오프)
- Auxon 구성 요소: 성능 데이터, 수요 예측, 리소스 공급, 리소스 가격, 의도 설정, 구성 언어 엔진, 솔버, 할당 계획
- "Stupid Solver" 접근: 완벽한 솔루션을 기다리지 않고 단순한 휴리스틱으로 시작하여 나중에 교체 - 솔버 인터페이스를 추상화하여 교체 용이하게 설계
- 불확실한 요구사항에 대한 대응: 소프트웨어를 범용적이고 모듈화되게 설계하여 변경 비용을 최소화
- 고객 확보: 기존 도구가 없는 팀을 먼저 타겟팅하고, 초기 성공 사례를 통해 내부 지지자를 확보
- 불가지론적(agnostic) 설계: "있는 그대로 오세요" - 특정 도구에 종속되지 않는 범용적 설계로 진입 장벽 최소화
- 100% 조직 채택을 성공으로 정의하지 않는다 - 장기 꼬리의 모든 서비스에 기능을 맞추는 것은 수확 체감
- 팀 구성: 제너럴리스트와 도메인 전문가를 결합하되, 전문가 투입 시기는 프로젝트가 성공적으로 궤도에 오른 후
- SRE 소프트웨어 개발자는 SRE로서의 업무를 계속 수행해야 한다 - 프로덕션 세계와의 연결 유지가 핵심

## 핵심 인용
> "Team size should not scale directly with service growth."

> "Don't focus on perfection and purity of solution, especially if the bounds of the problem aren't well known. Launch and iterate."

> "Specify the requirements, not the implementation."
