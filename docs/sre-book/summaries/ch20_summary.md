# Load Balancing in the Datacenter 요약

## 핵심 원칙
- 이상적인 경우 모든 백엔드 태스크의 부하가 완벽히 균등 분산되어 가장 적게 사용하는 태스크와 가장 많이 사용하는 태스크의 CPU 소비가 동일하다
- 열악한 데이터센터 내 로드 밸런싱은 리소스 가용성을 인위적으로 제한한다 - 1,000 CPU를 예약해도 실제로 700 CPU만 사용 가능할 수 있다
- 분산된 의사결정 환경에서 클라이언트는 부분적이고/또는 오래된 백엔드 상태 정보만으로 실시간 결정을 내려야 한다
- 랜덤 서브세팅은 부하를 매우 불균등하게 분산하므로, 결정론적 서브세팅이 Google의 해법이다
- 백엔드 제공 정보를 의사결정에 반영하는 가중치 라운드 로빈(Weighted Round Robin)이 가장 효과적인 정책이다

## 주요 프랙티스 및 권고사항
- 비정상 태스크 식별 - 흐름 제어: 클라이언트가 백엔드별 활성 요청 수를 추적하여 한계(보통 100)에 도달하면 전송 중단
- Lame Duck 상태: 백엔드가 명시적으로 클라이언트에게 새 요청 중단을 요청하여 진행 중인 요청은 정상 완료하고 깔끔하게 종료
- Lame Duck 전파: Google RPC에서 비활성 클라이언트도 주기적 UDP 헬스 체크를 보내므로 1-2 RTT 내에 모든 클라이언트에 전파
- 서브세팅: 각 클라이언트가 20-100개의 백엔드와 연결하여 메모리/CPU 오버헤드를 최소화하면서 충분한 부하 분산 확보
- 랜덤 서브세팅: 300 클라이언트, 300 백엔드, 30% 서브셋에서 최소 63%~최대 121% 부하 불균형 - 75% 이상 서브셋이 필요하여 비실용적
- 결정론적 서브세팅: 클라이언트를 "라운드"로 분할하고, 각 라운드 내에서 백엔드를 정확히 한 클라이언트에 할당 - 완벽한 균등 분배 달성
- 라운드별 다른 시드로 셔플: 백엔드 장애 시 해당 부하가 서브셋 내가 아닌 전체 백엔드에 분산되도록 보장
- 단순 라운드 로빈: 가장 간단하지만 최대 2배의 CPU 소비 차이 발생 - 소규모 서브셋, 쿼리 비용 변동, 머신 다양성, 예측 불가 성능 요인
- 최소 부하 라운드 로빈: 활성 요청이 가장 적은 백엔드에 요청 전송 - 하지만 비정상 태스크가 빠른 오류를 반환하면 트래픽을 "싱크홀"
- 싱크홀 방지: 최근 오류를 활성 요청처럼 카운트하여 비정상 태스크에서 부하를 전환
- 가중치 라운드 로빈: 각 백엔드가 응답에 현재 쿼리율, 오류율, 활용도를 포함하여 클라이언트가 능력 점수를 조정
- GCU(Google Compute Units): 다양한 CPU 아키텍처의 성능을 정규화하는 가상 단위로 머신 다양성 문제 해결
- 태스크 재시작 시 Lame Duck 상태로 프리워밍(prewarming)하여 성능이 정상화될 때까지 사용자 트래픽 차단

## 핵심 인용
> "Poor in-datacenter load balancing practices artificially limit resource availability."

> "If your load balancing policy can't adapt to unforeseen performance limitations, you will inherently end up with a suboptimal load distribution when working at scale."
