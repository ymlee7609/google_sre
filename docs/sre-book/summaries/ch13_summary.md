# Emergency Response 요약

## 핵심 원칙
- 시스템은 반드시 장애가 발생하며, 상상하지 못한 방식으로 장애가 발생한다
- 긴급 대응의 핵심: 당황하지 않기, 필요하면 더 많은 사람을 투입하기, 과거로부터 학습하기
- 세 가지 긴급 상황 유형: 테스트 유발, 변경 유발, 프로세스(자동화) 유발
- 장애 이론과 현실은 완전히 다른 영역이다 - 실제 장애 전까지는 시스템 반응을 알 수 없다
- 해결책은 항상 존재한다 - 혼자 찾지 못하면 범위를 넓혀라
- 가장 많은 상태(state)를 가진 사람이 종종 이벤트를 촉발한 사람이다 - 그 사람을 활용하라

## 주요 프랙티스 및 권고사항
- 테스트 유발 사례: MySQL 데이터베이스 접근 차단 테스트에서 예상치 못한 종속성이 발견됨 - 롤백 절차를 사전에 테스트 환경에서 검증해야 한다
- 변경 유발 사례: 금요일 글로벌 설정 변경이 전체 외부 시스템 크래시 루프를 유발 - 리스크 인식과 무관하게 철저한 카나리 프로세스가 필요하다
- 프로세스 유발 사례: Diskerase 자동화의 빈 집합 = "모든 것" 버그 - 자동화에 위생 검사와 속도 제한을 포함한다
- 장애 이력(outage history)을 철저히 기록하고, 모든 사람이 학습할 수 있도록 포스트모템을 게시한다
- 대규모 "What if" 질문을 정기적으로 던진다: 전원 장애, 네트워크 침수, 데이터센터 중단 등
- 사전 예방적 테스트(proactive testing)를 장려한다 - 통제된 상황에서 장애를 경험하는 것이 새벽 2시보다 낫다
- 대역 외(out-of-band) 통신 시스템을 유지하고 정기적으로 사용한다
- 인시던트 대응 프로세스를 수립하고, 정기적으로 연습하고, 업데이트를 명확히 전파한다
