# SRE와 카오스 엔지니어링: 통신사업자 관점의 적용 가이드

## 1. 카오스 엔지니어링 개요

### 1.1 정의와 철학

카오스 엔지니어링(Chaos Engineering)은 분산 시스템의 안정성과 복원력을 검증하기 위해 의도적으로 장애를 주입하는 실험 방법론이다. "장애는 반드시 발생한다"는 전제 하에, 통제된 환경에서 먼저 장애를 일으켜 시스템의 약점을 사전에 발견하는 것이 핵심이다.

### 1.2 작동 방식

카오스 엔지니어링은 네 단계의 실험 사이클로 구성된다.

1. **Steady State 정의**: 시스템의 정상 상태를 측정 가능한 지표로 정의
2. **가설 수립**: "이 장애가 발생해도 정상 상태가 유지될 것이다"라는 가설 설정
3. **장애 주입(Fault Injection)**: 서버 종료, 네트워크 지연, CPU 과부하 등 실제 장애 시뮬레이션
4. **관찰 및 개선**: 정상 상태 유지 여부를 관찰하고, 발견된 취약점을 개선

### 1.3 탄생과 발전

Netflix가 AWS로 마이그레이션하면서 2011년경 Chaos Monkey를 만든 것이 시초이다. 프로덕션 환경에서 무작위로 인스턴스를 종료시켜 서비스가 이를 견딜 수 있는지 테스트하였으며, 이후 Simian Army라는 도구 모음으로 확장되었다. 2015년 AWS 대규모 장애 시 Netflix가 사실상 무피해로 넘긴 것이 카오스 엔지니어링의 효과를 증명한 대표적 사례이다.

### 1.4 주요 도구

| 도구 | 특징 |
|------|------|
| Chaos Monkey / Simian Army | Netflix 개발, 카오스 엔지니어링의 원조 |
| Gremlin | 상용 Failure-as-a-Service 플랫폼 |
| LitmusChaos | CNCF 인큐베이팅 프로젝트, Kubernetes 네이티브 |
| Chaos Mesh | PingCAP 개발, Kubernetes 전용 오픈소스 |
| ChaosBlade | 알리바바 개발, 200+ 실험 시나리오 지원 |
| AWS Fault Injection Service | AWS 네이티브 장애 주입 서비스 |
| Spirent Landslide CNF Resiliency | 통신사 5G CNF 복원력 테스트 전용 |

### 1.5 통신사업자 적용 사례

통신사업자는 전통적으로 보수적인 접근을 취해왔으나, 5G SA와 클라우드 네이티브 전환을 계기로 카오스 엔지니어링 도입이 시작되고 있다.

- **Rakuten**: Chaos Mesh를 활용하여 Kubernetes 기반 인프라에서 Fault Injection, 네트워크 지연 시뮬레이션, 트래픽 급증 테스트 등을 적극 수행
- **Jio Platforms (인도)**: Spirent과 협력하여 1.9억 5G 가입자 대상 전국 5G SA 코어에 자동화된 zero-touch 테스팅 프레임워크 구축
- **미국 Tier 1 사업자**: Spirent과 클라우드 인프라 벤치마킹 평가 진행 중 (CNF 복원력 테스트의 전 단계)
- **China Mobile**: 알리바바의 ChaosBlade 플랫폼 활용
- **SNCF (프랑스)**: 2017년 "Day of Chaos" 이벤트로 게이미피케이션 기반 장애 시뮬레이션 수행

대부분의 통신사업자는 프로덕션이 아닌 **사전 프로덕션(pre-production) 또는 랩 환경**에서 CNF 복원력 테스트 형태로 도입하고 있으며, CI/CD 파이프라인에 통합하는 방향으로 발전 중이다.

## 2. SRE(Site Reliability Engineering) 개요

### 2.1 정의와 철학

SRE는 Google이 2003년경 창안한 개념으로, "소프트웨어 엔지니어에게 운영 문제를 맡기면 어떻게 될까?"라는 질문에서 출발한다. 운영을 엔지니어링 문제로 바라보고, 측정 가능한 지표와 자동화를 통해 신뢰성을 **관리**하는 데 초점을 둔다.

### 2.2 핵심 구성 요소

**신뢰성 지표 체계**

- **SLI (Service Level Indicator)**: 서비스 신뢰성을 측정하는 정량적 지표 (예: 응답 성공률, 레이턴시)
- **SLO (Service Level Objective)**: SLI에 대한 목표값 (예: 월간 가용성 99.95%)
- **SLA (Service Level Agreement)**: SLO를 기반으로 한 고객과의 계약 (미달 시 보상 조건 포함)
- **Error Budget**: SLO와 100% 사이의 차이로 정의되는 "허용 가능한 장애 예산"

**운영 프랙티스**

- **Toil 제거**: 반복적 수작업을 자동화로 대체
- **포스트모템(Postmortem)**: 비난 없는(blameless) 사후 분석 문화
- **온콜 관리**: 체계적인 당직 로테이션과 에스컬레이션
- **용량 계획(Capacity Planning)**: 수요 예측 기반 인프라 확장 계획

## 3. SRE vs 카오스 엔지니어링: 비교 분석

### 3.1 기본 철학 비교

SRE가 건물의 안전 기준을 설계하고 모니터링하는 **건축 감리사**라면, 카오스 엔지니어링은 실제로 지진을 시뮬레이션해보는 **내진 테스트**에 해당한다.

| 비교 항목 | SRE | 카오스 엔지니어링 |
|----------|-----|-----------------|
| **핵심 질문** | "얼마나 안정적인가?" | "정말 안정적인가?" |
| **접근 방식** | 모니터링, 알림, 자동화를 통한 관리 | 의도적 장애 주입을 통한 검증 |
| **활동 시점** | 상시 운영 (24/7) | 주기적 실험 (GameDay 등) |
| **주요 산출물** | SLO 대시보드, Error Budget 리포트, 런북 | 취약점 발견 보고서, 개선 권고 |
| **조직 형태** | 전담 SRE 팀 | SRE 팀 내 수행 또는 별도 팀과 협업 |
| **장애 대응 성격** | 사후 대응 + 예방적 자동화 | 사전 예방적 검증 |
| **기원** | Google (2003년경) | Netflix (2011년경) |

### 3.2 상호 보완 관계

SRE와 카오스 엔지니어링은 경쟁 관계가 아니라 상호 보완적이며, 실무에서는 SRE 프랙티스 위에 카오스 엔지니어링이 올라가는 구조가 일반적이다.

**SRE → 카오스 엔지니어링 방향**: SRE가 정의한 SLO가 카오스 실험의 성공/실패 판단 기준이 된다. 예를 들어, "AMF Pod를 강제 종료해도 SLO 99.95%가 유지되는가?"라는 카오스 실험은 SRE의 SLO 정의 없이는 판단 자체가 불가능하다.

**카오스 엔지니어링 → SRE 방향**: 카오스 실험에서 발견된 취약점이 SRE의 자동화 개선, 런북(Runbook) 업데이트, 모니터링 강화로 이어진다.

**Error Budget 연동**: Error Budget이 남아있을 때 카오스 실험을 수행하고, 소진되면 실험을 중단하고 안정화에 집중하는 방식으로 운용할 수 있다.

### 3.3 통신사업자 관점의 도입 순서

TM Forum의 자율운영 네트워크 레벨과 매핑하면 다음과 같은 진화 경로가 자연스럽다.

1. **Level 2 (부분 자동화)**: SRE 문화 도입 시작 — SLI/SLO 정의, 모니터링 자동화
2. **Level 3 (조건부 자동화)**: SRE 프랙티스 정착 — Error Budget 운용, Toil 제거, 포스트모템 정착
3. **Level 3~4 전환기**: 카오스 엔지니어링 도입 — self-healing 검증, CNF 복원력 테스트
4. **Level 4 (고도 자동화)**: 카오스 엔지니어링이 CI/CD에 내재화, 자율 복구 검증 상시화

기존 NOC 중심 운영에서 SLI/SLO 기반 운영으로의 전환이 선행되어야 하며, Deutsche Telekom의 사례에서 강조된 "코드를 쓸 수 있는 새로운 세대의 네트워크 엔지니어" 양성이 이 전환의 핵심 전제 조건이다.

## 4. Error Budget 심화: 개념과 작동 원리

### 4.1 Error Budget의 핵심 메커니즘

Error Budget은 SLO와 100% 사이의 차이로 정의되며, **개발 속도와 안정성 사이의 트레이드오프를 정량적으로 관리**하는 도구이다.

**계산 예시**: SLO가 99.95%이면 월간 허용 다운타임은 약 21.6분이다.

| SLO | 월간 Error Budget | 연간 Error Budget |
|-----|------------------|------------------|
| 99.9% | 약 43.2분 | 약 8.76시간 |
| 99.95% | 약 21.6분 | 약 4.38시간 |
| 99.99% | 약 4.3분 | 약 52.6분 |
| 99.999% | 약 26초 | 약 5.3분 |

**운용 정책**: Error Budget 잔여량에 따라 변경(배포, 실험) 속도를 조절한다.

- 잔여량 충분 → 새로운 변경 적극 허용
- 잔여량 부족 → 변경 속도 감소 또는 동결
- 완전 소진 → 긴급 패치 외 모든 변경 중단, 안정화 집중

### 4.2 Error Budget이 해결하는 조직적 문제

Error Budget의 진정한 가치는 기술적 도구를 넘어 **조직 갈등의 정량적 해소**에 있다. 개발팀("빨리 배포하고 싶다")과 운영팀("안정성을 지키고 싶다") 사이의 갈등을 감정이 아닌 수치로 판단하게 해준다.

## 5. Error Budget의 IT/NW 운영 적용: 비교 분석

### 5.1 적용이 달라져야 하는 근본 이유

두 영역에서 Error Budget 적용이 달라야 하는 핵심 이유는 **"변경-장애 인과관계의 강도"** 차이에 있다.

- **IT 운영**: 장애의 60~70%가 코드 배포, 설정 변경, DB 마이그레이션 등 자체 변경에서 비롯. Error Budget 소진 시 변경을 멈추면 장애가 줄어드는 인과 루프가 명확
- **NW 운영**: 광케이블 절단, 장비 HW 결함, 자연재해, 전원 장애 등 외부/물리적 요인의 비중이 큼. Error Budget 소진 시 변경 동결이 장애 감소로 직결되지 않음

### 5.2 IT 운영에서의 Error Budget 적용

IT 영역(BSS/OSS, 포탈, 프로비저닝 시스템 등)에서는 Google SRE 모델에 가깝게 적용할 수 있다.

**SLO 정의**: 서비스 단위로 깔끔하게 정의 가능

- 프로비저닝 시스템: API 성공률 99.95%, P99 응답시간 500ms 이내
- 고객 포탈: 페이지 로드 성공률 99.9%, 로그인 성공률 99.95%

**Error Budget 소비 정책**: 잔여량에 따른 단계적 배포 정책 적용

- 잔여량 50% 이상: 일반 배포 허용
- 잔여량 20~50%: 카나리 배포만 허용
- 잔여량 20% 미만: 긴급 패치 외 변경 동결

**주요 특성**

- Budget 구조: 단일 풀로 운영 가능
- Budget 윈도우: 주 또는 월 단위 리셋
- Budget 단위: 시간 기반 (분, 시간)
- 롤백 용이성: 높음 (컨테이너 이미지 롤백, DB 마이그레이션 롤백)
- 측정 자동화: 높음 (APM, Synthetic Monitoring)
- 주요 소비자: 개발팀 (배포)
- 주요 보호자: SRE/플랫폼팀

### 5.3 NW 운영에서의 Error Budget 적용

NW 영역에서는 원래 모델을 상당히 재설계해야 하며, 세 가지 핵심 변형이 필요하다.

#### 변형 1: 분리 회계 (Separated Budget Accounting)

NW 운영에서 가장 중요한 설계 포인트로, Error Budget을 단일 풀이 아닌 최소 두 개의 버킷으로 분리해야 한다.

**통제 가능 버킷 (Controllable Budget)**

- 대상: 소프트웨어 업그레이드, 설정 변경, 계획된 절체 작업 등 자체 유발 장애
- 대응: 전통적 Error Budget 메커니즘 적용 (소진 시 변경 동결)

**통제 불가 버킷 (Environmental Budget)**

- 대상: 광케이블 절단, 자연재해, 전원 장애, 장비 HW 결함
- 대응: 변경 동결이 아닌 **복원력 투자 트리거**로 활용
  - 지속적 초과 구간 → 이중화 경로 추가, 물리적 루트 다변화 등 인프라 투자 근거

분리하지 않을 경우의 문제: "이번 달 광케이블이 두 번 잘려서 Error Budget을 다 썼으니 보안 패치 펌웨어 업그레이드를 다음 달로 미루자"와 같은 비합리적 의사결정이 발생하며, 오히려 리스크가 증가하는 역설적 상황이 초래됨

#### 변형 2: 영향 규모 가중 (Customer-Impact Weighting)

NW에서는 동일한 "1건"의 장애라도 영향 범위가 극심하게 다르다. 코어 라우터 장애(수만 고객 영향)와 액세스 스위치 장애(수십 고객 영향)를 동일하게 취급할 수 없다.

- 단순 시간 기반(분/시간)이 아닌 **"고객·분(customer-minutes)"** 단위로 Error Budget 계산
- 예시: SLO를 "월간 고객·분 손실 10,000 이하"로 설정
  - 1만 명 × 1분 = 10,000 고객·분 (코어 장애)
  - 10명 × 100분 = 1,000 고객·분 (액세스 장애)
  - → 두 장애의 영향이 정량적으로 구분됨

#### 변형 3: Budget 윈도우 조정

IT에서의 주/월 단위 리셋과 달리, NW 대규모 변경은 분기에 한두 번 수준이므로 윈도우 불일치가 발생한다.

- 기본 윈도우: **분기 단위**가 NW 변경 주기에 적합
- 대규모 변경 시: 별도의 **임시 버짓(Change Budget)** 할당
  - 예: "코어 라우터 OS 업그레이드를 위해 추가 Error Budget 30분 할당"

**NW 적용 시 주요 특성 요약**

- SLO 단위: 계층별 + 서비스 경로별 (가용성, 패킷 손실, 지터의 다차원 SLO)
- Budget 구조: 통제 가능/불가 분리 회계
- Budget 윈도우: 분기 단위 + 대규모 변경 시 임시 할당
- Budget 단위: 고객·분 (customer-minutes) 가중
- 롤백 용이성: 낮음 (물리 절체, 라우팅 수렴 시간 소요)
- 측정 자동화: 중간 (SNMP/Streaming Telemetry 수집 후 가공 필요)
- 소진 시 대응: 통제 가능 버킷은 변경 동결 / 통제 불가 버킷은 복원력 투자 트리거
- 주요 소비자: 네트워크 엔지니어링팀 (설정 변경, 업그레이드)
- 주요 보호자: NOC + 네트워크 설계팀

### 5.4 IT/NW 적용 비교 종합표

| 설계 요소 | IT 운영 | NW 운영 |
|----------|---------|---------|
| **SLO 단위** | 서비스별 (API 성공률, 응답시간) | 계층별 + 서비스 경로별 (가용성, 패킷 손실, 지터) |
| **Budget 구조** | 단일 풀 | 통제 가능/불가 분리 회계 |
| **Budget 윈도우** | 주 또는 월 단위 | 분기 단위 + 대규모 변경 시 임시 할당 |
| **Budget 단위** | 시간 기반 (분, 시간) | 고객·분 (customer-minutes) 가중 |
| **소진 시 대응** | 변경 동결 (배포 중단) | 통제 가능: 변경 동결 / 통제 불가: 복원력 투자 트리거 |
| **주요 소비자** | 개발팀 (배포) | NW 엔지니어링팀 (설정 변경, 업그레이드) |
| **주요 보호자** | SRE / 플랫폼팀 | NOC + NW 설계팀 |
| **롤백 용이성** | 높음 (컨테이너, DB 롤백) | 낮음 (물리 절체, 라우팅 수렴) |
| **측정 자동화** | 높음 (APM, Synthetic) | 중간 (SNMP/Telemetry → 가공) |
| **변경-장애 인과관계** | 강함 (자체 변경이 장애 주원인) | 약함 (외부/물리적 요인 비중 큼) |

### 5.5 두 영역의 수렴 지점

SDN 컨트롤러, 네트워크 자동화 플랫폼, vCPE/vBNG 같은 경계 영역에서는 NW 기능이면서 IT 방식으로 운영되므로, 하이브리드 적용이 필요하다.

**SDN 컨트롤러 적용 예시**

- 컨트롤러 소프트웨어 가용성 → IT식 관리 (API 성공률, 배포 주기 연동)
- 플로우 룰 변경의 데이터 플레인 영향 → NW식 관리 (고객·분 기반, 분리 회계)

## 6. 결론 및 시사점

### Error Budget 적용의 핵심 원칙

1. **"허용 가능한 불완전성의 명시적 정의"** 라는 철학은 IT/NW 모두에 유효하다. 기존의 암묵적 "장애 제로" 목표는 비현실적이며, 오히려 변경 회피와 기술 부채 누적을 초래한다.

2. **구현 방식은 대상 시스템의 물리적 특성에 맞게 재설계**되어야 한다. IT 운영에서는 Google SRE 원래 모델을 거의 그대로 적용 가능하며, NW 운영에서는 분리 회계, 가중치 적용, 윈도우 조정이라는 세 가지 핵심 변형이 필수적이다.

3. **SRE 문화의 선행 정착**이 전제 조건이다. SLI/SLO가 정의되어 있어야 Error Budget이 작동하고, Error Budget이 작동해야 카오스 엔지니어링의 성공/실패 기준이 성립한다.

4. **SDN/NFV 전환 영역**에서 IT식과 NW식의 하이브리드 적용을 통해 점진적으로 확대해가는 것이 현실적인 도입 전략이다.
