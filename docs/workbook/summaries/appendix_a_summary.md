# Example SLO Document 요약

## 핵심 원칙
- SLO 문서는 서비스 개요, SLI/SLO 정의, 근거, 에러 버짓, 주의사항을 포함하는 공식 문서이다
- 각 SLO 목표는 별도의 에러 버짓을 가진다 (100% - 목표 = 에러 버짓)
- SLO 수치는 실제 측정 데이터에 기반하여 설정하되, 증거 기반이 아닌 경우에도 이 사실을 문서화해야 한다
- SLO 문서에는 작성자, 리뷰어, 승인자, 재검토 일자가 명시되어야 한다
- 롤링 윈도우(예: 4주)를 기준으로 SLO를 측정한다

## 주요 프랙티스 및 권고사항
- **서비스 유형별 SLI/SLO 정의**:
  - **API 서버**: 가용성(5XX 비율 기준 97%), 지연시간(90% < 400ms, 99% < 850ms)
  - **HTTP 서버**: 가용성(5XX 비율 기준 99%), 지연시간(90% < 200ms, 99% < 1000ms)
  - **스코어 파이프라인**: 신선도(90% < 1분, 99% < 10분), 정확성(99.99999%), 완전성(99% 완전 처리)
- **SLI 측정 위치**: 로드 밸런서 메트릭 사용 - 사용자 요청이 로드 밸런서에 도달하지 못한 경우 정확히 측정하지 못할 수 있음을 명시
- **에러 정의**: HTTP 5XX 상태 코드만 에러로 분류, 그 외 모든 것은 성공으로 간주
- **정확성 검증**: 합성 데이터를 주입하는 정확성 프로버(correctness prober)를 사용하여 올바른 출력 검증
- **SLO 수치 결정 방법**: 실제 운영 데이터 기반으로 가용성은 가장 가까운 1%로 내림, 지연시간은 가장 가까운 50ms로 올림
- **에러 버짓 계산 예시**: API 서버 97% SLO에서 4주간 100만 요청이면, 에러 버짓은 3만 에러
- **에러 버짓 정책 연동**: 에러 버짓 소진 시 에러 버짓 정책(별도 문서)을 발동한다

## 핵심 인용
> "Even if the numbers in the SLO are not strongly evidence-based, it is necessary to document this so that future readers can understand this fact, and make their decisions appropriately."
