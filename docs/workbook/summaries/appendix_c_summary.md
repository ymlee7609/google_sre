# Results of Postmortem Analysis 요약

## 핵심 원칙
- Google은 표준화된 포스트모템 템플릿을 사용하여 인시던트의 근본 원인과 트리거를 일관되게 캡처한다
- 트렌드 분석을 통해 체계적인 근본 원인 유형(소프트웨어 인터페이스 설계 결함, 미성숙한 배포 계획 등)을 대상으로 개선한다
- 장애의 68%가 바이너리 푸시(37%)와 설정 푸시(31%)에 의해 트리거된다
- 소프트웨어가 장애 근본 원인의 41.35%를 차지하며, 개발 프로세스 실패가 20.23%로 뒤를 잇는다
- 하드웨어 장애는 전체 장애 트리거의 2%에 불과하다

## 주요 프랙티스 및 권고사항
- **상위 8대 장애 트리거** (2010-2017, 수천 건의 포스트모템 분석):
  - 바이너리 푸시: 37%
  - 설정 푸시: 31%
  - 사용자 행동 변화: 9%
  - 처리 파이프라인: 6%
  - 서비스 제공자 변경: 5%
  - 성능 저하: 5%
  - 용량 관리: 5%
  - 하드웨어: 2%
- **상위 5대 근본 원인 카테고리**:
  - 소프트웨어: 41.35%
  - 개발 프로세스 실패: 20.23%
  - 복잡한 시스템 동작: 16.90%
  - 배포 계획: 6.74%
  - 네트워크 장애: 2.75%
- **실무 시사점**:
  - 바이너리와 설정 변경이 장애의 최대 원인이므로, 카나리 배포와 점진적 롤아웃이 필수적이다
  - 소프트웨어 결함 비중이 높으므로 코드 리뷰, 테스트, 정적 분석에 투자하라
  - 개발 프로세스 실패(20.23%)는 더 나은 변경 관리, 리뷰 프로세스, 자동화로 해결 가능하다
  - 복잡한 시스템 동작(16.90%)은 시스템 단순화와 장애 모드 이해에 투자해야 함을 시사한다
  - 하드웨어 장애(2%)가 매우 낮다는 것은 소프트웨어 계층의 중복성/복원력이 효과적임을 보여준다
  - 포스트모템 데이터의 트렌드 분석을 통해 투자 우선순위를 데이터 기반으로 결정하라

## 핵심 인용
> 데이터 자체가 핵심 메시지이다: 장애의 68%는 변경(바이너리 + 설정 푸시)에 의해 트리거되며, 근본 원인의 61.58%는 소프트웨어와 개발 프로세스 실패에 기인한다.
