# Investigating Systems 요약

## 핵심 원칙
- 대부분의 시스템은 결국 실패하며, 복잡한 시스템을 조사하는 능력은 적절한 로그, 정보 소스, 전문성에 의존한다
- 디버깅과 보안 조사는 유사하지만 중요한 차이점이 있다: 보안 조사에서는 적대자가 존재한다
- 로깅 시스템 자체도 보호와 접근 제어를 고려하여 설계해야 한다
- 디버깅 접근 권한은 읽기 전용이라도 남용 위험이 있으므로 적절한 보안 메커니즘이 필요하다
- 로그 보존 기간과 상세도 간의 트레이드오프를 신중히 결정해야 한다
- 로그에는 민감 정보(PII, 인증 정보)가 포함되지 않아야 하며, 로그 자체가 공격 대상이 될 수 있다
- 불변 로그(immutable log)는 포렌식 조사에 필수적이다
- 디버깅에 막혔을 때의 전략(가설 세우기, 이분법 탐색, 시스템 직접 관찰 등)이 중요하다

## 주요 프랙티스 및 권고사항
- 구조화된 로깅을 사용하여 검색과 분석을 용이하게 할 것
- 로그 접근에 최소 권한 원칙을 적용하고, 접근 감사를 수행할 것
- 보안 관련 로그는 별도로 보호하고, 변조 방지 메커니즘을 적용할 것
- 로그에 상관 ID(correlation ID)를 포함하여 분산 시스템의 요청 추적을 가능하게 할 것
- 디버깅과 보안 조사를 구분하고, 보안 조사에서는 정보 공유를 제한할 것 (need-to-know)
- 비상 시 로그 접근을 위한 브레이크글라스 메커니즘을 마련할 것
- 장애 복구 후 사후 분석(postmortem)을 수행하고, 비난 없는 문화를 조성할 것
- 프로덕션 디버깅 도구를 사전에 테스트하고 유지보수할 것

## 핵심 인용
> "적절한 로그와 정보 소스에 접근할 수 있는 것 외에도, 전문성이 필요하다. 또한 로깅 시스템을 보호와 접근 제어를 염두에 두고 설계해야 한다."

> "읽기 전용 디버깅 접근 권한을 부여하는 것조차 이 접근이 남용될 수 있는 위험을 만든다. 이 위험을 해결하기 위해 적절한 보안 메커니즘이 필요하다."

> "보안 조사에서는 적대자가 복구 노력에 대해 알아차리지 못하도록 정보를 need-to-know 기반으로 공유해야 한다."
