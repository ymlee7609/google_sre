# Design for Recovery 요약

## 핵심 원칙
- 모든 복잡한 시스템은 신뢰성 문제를 경험하며, 공격자는 명시적으로 시스템 장애를 유발하려 한다
- 제품 개발 초기 단계에서 장애를 예측하고 복구 프로세스를 계획해야 한다
- 복구 메커니즘은 정기적으로 실행되어 실제로 작동하는지 확인해야 한다
- 롤백은 보안 취약점을 재도입할 수 있으므로, 복구와 보안 간의 트레이드오프를 신중히 판단해야 한다
- 시스템이 돌아가야 할 "알려진 양호한 상태(known good state)"를 명확히 정의해야 한다
- 속도 제한(rate limiting)은 장애의 확산을 방지하는 핵심 메커니즘이다
- 버전 번호 체계를 통해 상태 관리와 롤백을 체계적으로 수행할 수 있다
- 폐기(revocation) 메커니즘은 침해된 자격증명이나 인증서를 무효화하는 데 필수적이다
- 복구 속도와 복구 안전성 사이의 균형이 필요하다

## 주요 프랙티스 및 권고사항
- 모든 변경에 대해 롤백 계획을 수립하고, 롤백이 보안 취약점을 재도입하지 않는지 확인할 것
- 점진적 롤아웃과 카나리 배포를 통해 변경의 폭발 반경을 제한할 것
- 암호화 키의 정기적 순환(rotation)을 자동화할 것
- 인증서 폐기 메커니즘(CRL, OCSP)의 신뢰성을 확보할 것
- 복구 절차를 정기적으로 훈련하고 테스트할 것 (연습하지 않은 복구는 실패한다)
- 비상 접근 메커니즘(브레이크글라스)을 마련하되, 남용을 방지하는 감사 체계를 구축할 것
- 시스템 상태의 불변 로그를 유지하여 감사 추적(audit trail)을 보장할 것
- 자동 복구와 수동 개입의 경계를 명확히 정의할 것

## 핵심 인용
> "모든 복잡한 시스템은 신뢰성 문제를 경험하며, 공격자는 명시적으로 시스템을 장애시키거나 의도된 기능에서 벗어나게 만들려 한다."

> "복구 메커니즘이 정기적으로 실행되지 않으면, 정작 필요할 때 작동하지 않을 가능성이 높다."

> "롤백은 시스템을 건강한 상태로 되돌릴 수 있지만, 동시에 이미 수정된 보안 취약점을 재도입할 수도 있다."

## 관련 키워드
복구 설계, 롤백, 속도 제한, 버전 관리, 키 순환, 인증서 폐기, CRL, OCSP, 브레이크글라스, 점진적 롤아웃, 카나리 배포, 알려진 양호한 상태, 감사 추적, 자동 복구
