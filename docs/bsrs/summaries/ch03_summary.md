# Case Study: Safe Proxies 요약

## 핵심 원칙
- Safe Proxy는 프로덕션 환경에서 권한 있는 관리자가 실수나 악의로 문제를 일으킬 가능성을 제한하는 프레임워크이다
- Zero Touch Prod 원칙: 프로덕션의 모든 변경은 자동화에 의해 수행되거나, 소프트웨어로 사전 검증되거나, 감사된 브레이크글라스 메커니즘을 통해 트리거되어야 한다
- 프록시는 기존 시스템에 대한 대규모 변경 없이 새로운 보안/신뢰성 요구사항을 충족시키는 방법이다
- 클라이언트는 대상 시스템에 직접 접근하지 않고 프록시를 통해서만 접근한다
- 모든 작업은 중앙에서 감사(audit), 접근 제어, 속도 제한이 가능하다
- Google은 약 13%의 장애가 Zero Touch Prod로 예방 또는 완화될 수 있었다고 추정한다
- 프록시는 다자 승인(MPA)을 강제하는 중앙 집중 지점 역할을 한다
- 브레이크글라스 메커니즘을 통해 긴급 상황에서의 직접 접근을 허용한다

## 주요 프랙티스 및 권고사항
- Safe Proxy를 통해 관리 작업의 모든 감사 로그를 중앙 집중화할 것
- ACL 기반의 세밀한 접근 제어 정책을 구성할 것
- 위험한 명령(서버 재시작, 작업 종료 등)에 대해 MPA를 요구할 것
- 속도 제한을 통해 변경의 폭발 반경(blast radius)을 제한할 것
- 프록시의 단일 장애점(SPOF) 문제는 다중 인스턴스로 완화할 것
- 정책 설정의 오류를 방지하기 위해 안전한 기본값 템플릿을 제공할 것
- Tool Proxy를 사용하여 CLI 도구의 직접 실행을 차단하고 프록시를 통해서만 실행하도록 할 것
- 사용자 저항을 줄이기 위해 프록시가 기존 API와 동일한 인터페이스를 제공하도록 설계할 것

## 핵심 인용
> "Zero Touch Prod는 프로덕션의 모든 변경이 자동화에 의해 수행되거나, 소프트웨어로 사전 검증되거나, 감사된 브레이크글라스 메커니즘을 통해 트리거되도록 요구한다."

> "Safe Proxy를 사용하면 감사, 접근 제어, 속도 제한을 중앙에서 관리할 수 있으며, Google 평가 장애의 약 13%를 예방하거나 완화할 수 있었을 것으로 추정한다."

> "프록시는 기존 시스템에 대한 대규모 변경 없이 새로운 보안 및 신뢰성 요구사항을 충족시킬 수 있는 비용 효율적인 방법이다."
