# Testing Code 요약

## 핵심 원칙
- 아무리 주의 깊은 개발자라도 실수와 간과된 엣지 케이스는 불가피하며, 테스트는 장애 위험을 완화하는 핵심 수단이다
- 테스트는 단위 테스트, 통합 테스트, 정적 분석, 동적 분석, 퍼즈 테스트 등 다양한 계층으로 구성해야 한다
- 보안 테스트와 신뢰성 테스트는 서로 보완적이며, 둘 다 소프트웨어 개발 수명주기에 통합되어야 한다
- 테스트 피라미드: 빠르고 격리된 단위 테스트를 많이, 느리고 비용이 높은 E2E 테스트를 적게 구성한다
- 정적 분석은 코드를 실행하지 않고 버그와 취약점을 탐지하며, 개발자 워크플로에 통합되어야 한다
- 퍼즈 테스트는 예상치 못한 입력을 자동 생성하여 크래시와 보안 취약점을 발견한다
- 코드 커버리지는 유용한 지표이지만, 높은 커버리지가 높은 품질을 보장하지는 않는다
- 테스트 자체도 유지보수되어야 하며, 깨진 테스트를 방치하면 테스트 스위트의 가치가 떨어진다

## 주요 프랙티스 및 권고사항
- 모든 코드 변경에 대해 단위 테스트를 작성하고, CI 파이프라인에서 자동 실행할 것
- 통합 테스트로 구성 요소 간의 상호작용을 검증할 것
- Tricorder 같은 정적 분석 도구를 코드 리뷰 프로세스에 통합할 것
- AddressSanitizer(ASAN), ThreadSanitizer(TSAN) 등 동적 분석 도구를 활용할 것
- 보안에 민감한 코드에 대해 퍼즈 테스트를 수행할 것 (coverage-guided fuzzing)
- 부정적 테스트 케이스(잘못된 입력, 권한 없는 접근 등)를 명시적으로 작성할 것
- 성능 테스트와 부하 테스트로 과부하 상황의 동작을 검증할 것
- Red team 연습을 통해 실제 공격 시나리오에 대한 방어를 검증할 것

## 핵심 인용
> "신뢰할 수 있는 시스템은 장애에 대한 복원력이 있고 문서화된 서비스 수준 목표를 충족하며, 이는 보안 보장도 포함할 수 있다."

> "퍼즈 테스트는 비정상적인 입력을 자동으로 생성하여 개발자가 예상하지 못한 크래시와 보안 취약점을 발견하는 데 특히 효과적이다."

> "정적 분석의 가장 효과적인 배포 방식은 개발자의 워크플로에 직접 통합하는 것이다. 별도의 도구로 실행하면 개발자가 무시하기 쉽다."
