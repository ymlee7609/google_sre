# Deploying Code 요약

## 핵심 원칙
- 프로덕션에 실행 중인 코드가 실제로 의도한 코드인지 확인할 수 있어야 한다
- 소프트웨어 공급망(supply chain) 전체에 걸쳐 위협 모델을 정의하고 보호해야 한다
- 배포 자체가 시스템에 대한 변경이므로, 모든 변경은 보안 및 신뢰성 위험을 수반한다
- **검증 가능한 빌드(verifiable builds)**: 빌드 프로세스가 재현 가능하고, 빌드 아티팩트의 출처를 증명할 수 있어야 한다
- **출처 기반 배포 정책(provenance-based deployment policy)**: 코드의 출처와 빌드 과정을 검증하여 배포를 허가한다
- 코드 리뷰, 자동 테스트, 빌드 검증은 배포 전 안전성을 확보하는 다층 방어이다
- 점진적 롤아웃과 카나리 배포는 배포 위험을 제한하는 핵심 메커니즘이다
- 내부자 위협과 공급망 공격에 대한 방어가 배포 파이프라인에 내장되어야 한다

## 주요 프랙티스 및 권고사항
- 바이너리 인가(Binary Authorization) 정책을 구현하여 승인된 코드만 프로덕션에 배포할 것
- 밀폐 빌드(hermetic build)를 사용하여 빌드 재현성을 보장할 것
- 코드 리뷰를 필수화하고, 보안에 민감한 코드는 추가 리뷰어를 요구할 것
- 배포 파이프라인에 자동화된 보안 검사를 통합할 것
- 카나리 배포로 소규모 사용자에게 먼저 변경을 적용한 후 점진적으로 확대할 것
- 롤백 메커니즘을 항상 준비하고, 빠른 롤백이 가능하도록 설계할 것
- 설정(configuration) 변경도 코드 변경과 동일한 검증 프로세스를 적용할 것
- 제3자 의존성의 무결성을 검증하고, 의존성 업데이트를 자동 추적할 것

## 핵심 인용
> "프로덕션 환경에서 실행 중인 코드가 당신이 가정하는 코드인가? 안전하지 않은 배포를 방지하거나 탐지하기 위한 제어가 시스템에 필요하다."

> "배포된 아티팩트를 순수하게 검사하는 것만으로는 그것이 안전한지 판단하기 어렵다. 소프트웨어 공급망의 다양한 단계에 대한 제어가 아티팩트의 안전성에 대한 확신을 높일 수 있다."

> "코드 리뷰는 실수의 가능성을 줄이고 적대자가 악의적인 변경을 하는 것을 억제하며, 자동화된 테스트는 코드가 올바르게 작동한다는 확신을 높인다."
