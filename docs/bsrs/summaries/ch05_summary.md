# Design for Least Privilege 요약

## 핵심 원칙
- **최소 권한 원칙**: 사용자와 시스템은 작업 수행에 필요한 최소한의 접근 권한만 가져야 한다
- 불필요한 권한은 실수, 버그, 침해의 공격 표면을 확대하며, 실행 중인 시스템에서 제한하기 비용이 많이 든다
- 접근 권한은 위험도에 따라 분류하여 관리해야 한다 (읽기 전용 < 쓰기 < 관리자)
- 권한은 개발 수명주기 초기, 새로운 기능의 설계 단계에서 추가하는 것이 가장 효과적이다
- **제로 트러스트 네트워킹**: 네트워크 위치만으로 신뢰를 부여하지 않고, 모든 접근에 인증과 권한 부여를 요구한다
- 임시 접근 권한은 영구 권한보다 안전하며, 접근이 필요한 시간 동안만 부여해야 한다
- 다자 승인(MPA)은 중요한 작업에 대한 추가 안전장치 역할을 한다
- 사람에 대한 접근 제어뿐 아니라 서비스 간 통신에도 최소 권한을 적용해야 한다

## 주요 프랙티스 및 권고사항
- 소규모 기능적 API를 설계하여 세밀한 접근 제어를 가능하게 할 것
- BeyondCorp 모델처럼 기기 상태와 사용자 인증 기반의 제로 트러스트 접근을 구현할 것
- 관리자 접근에는 시간 제한이 있는 임시 토큰을 사용할 것
- 민감한 작업에는 다자 승인(MPA)을 요구하고, 비즈니스 정당성(Business Justification)을 기록할 것
- 브레이크글라스 메커니즘을 통해 긴급 상황에서 권한 상승을 허용하되, 모든 접근을 감사 로그에 기록할 것
- 서비스 계정에는 역할 기반 접근 제어(RBAC)를 적용할 것
- 설정 배포 시에도 최소 권한을 적용하여 위험을 분산할 것
- 프로덕션 접근 권한과 개발 접근 권한을 명확히 분리할 것

## 핵심 인용
> "최소 권한 원칙은 사용자가 현재 작업을 수행하는 데 필요한 최소한의 접근 권한을 가져야 한다고 규정한다."

> "불필요한 권한은 증가하는 공격 표면으로 이어지며, 실행 중인 시스템에서 보안 및 신뢰성 위험을 억제하거나 최소화하는 데 비용이 많이 든다."

> "긴급 상황에서는 브레이크글라스 메커니즘을 통해 정상적인 접근 제어를 우회할 수 있어야 하지만, 모든 브레이크글라스 접근은 감사되고 사후 검토되어야 한다."

## 관련 키워드
