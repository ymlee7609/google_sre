# Design for a Changing Landscape 요약

## 핵심 원칙
- "변화만이 유일한 상수"이며, 소프트웨어 시스템은 변화에 적응할 수 있도록 설계해야 한다
- 자주 빌드하고, 자동화된 테스트를 통한 릴리스를 수행하며, 컨테이너와 마이크로서비스를 활용해야 한다
- 의존성 관리가 보안의 핵심이며, 취약한 라이브러리는 신속하게 업데이트해야 한다
- 대부분의 설계 결정은 시스템 설계 단계에서 가장 쉽고 저렴하게 구현할 수 있다
- 컨테이너와 마이크로서비스 아키텍처는 격리와 독립적 배포를 가능하게 한다
- 짧은 릴리스 주기가 변경 위험을 줄이고, 문제 발생 시 빠른 롤백을 가능하게 한다
- 보안 취약점에 대한 빠른 패치 적용 능력이 조직의 보안 수준을 결정한다
- 인프라의 변경(암호화 알고리즘, 인증 프로토콜 등)도 계획적으로 수행해야 한다

## 주요 프랙티스 및 권고사항
- 의존성을 자동으로 추적하고, 알려진 취약점이 있는 버전을 탐지하는 도구를 구축할 것
- 컨테이너를 사용하여 애플리케이션을 격리하고, 기본 이미지를 정기적으로 업데이트할 것
- 자동 빌드 및 테스트 파이프라인을 구축하여 지속적 통합을 실현할 것
- 카나리 릴리스와 점진적 롤아웃으로 변경의 폭발 반경을 제한할 것
- HTTPS 마이그레이션처럼 장기적인 인프라 전환을 계획적으로 추진할 것
- 암호화 민첩성(cryptographic agility)을 설계에 내장하여 알고리즘 교체를 용이하게 할 것
- 서비스 메시(service mesh)를 활용하여 서비스 간 통신의 보안과 관찰성을 확보할 것
- 프로덕션 환경의 변경은 항상 롤백 가능하도록 설계할 것

## 핵심 인용
> "취약점이 발견된 후 패치를 적용하는 데 걸리는 시간이 조직의 보안 수준을 결정한다. 빠른 릴리스 주기를 가진 조직이 더 안전하다."

> "컨테이너는 애플리케이션을 격리하고, 재현 가능한 빌드를 가능하게 하며, 인프라 전반에 걸쳐 일관된 보안 정책을 적용할 수 있게 한다."

> "변화에 대한 설계는 새로운 취약점과 공격 시나리오에 신속하게 대응할 수 있을 뿐 아니라, 변화하는 비즈니스 요구사항도 더 빠르게 수용할 수 있게 한다."
