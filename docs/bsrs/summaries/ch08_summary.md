# Design for Resilience 요약

## 핵심 원칙
- **복원력(resilience)**: 공격에 대한 방어와 비정상적 상황에서 시스템의 신뢰성을 유지하는 능력
- 설계 초기에 여러 동시 인시던트 상황에서도 시스템을 전체 또는 부분적으로 운영하는 방법을 계획해야 한다
- **방어 심층(Defense in Depth)**: 다중의 독립적 방어 메커니즘을 적용하여 단일 장애점을 제거한다
- **폭발 반경(Blast Radius) 제어**: 장애나 침해의 영향 범위를 제한하는 설계가 필수적이다
- **장애 도메인(Failure Domain) 분리**: 시스템을 독립된 장애 영역으로 분리하여 전파를 차단한다
- **제어된 성능 저하(Graceful Degradation)**: 장애 시 핵심 기능을 우선 유지하며 비핵심 기능을 점진적으로 축소한다
- 지속적 검증(Continuous Validation)을 통해 복원력 가정을 주기적으로 확인해야 한다
- 카오스 엔지니어링을 통해 실제 장애 상황을 시뮬레이션하고 약점을 발견해야 한다

## 주요 프랙티스 및 권고사항
- Google App Engine 사례처럼 다층적 보안 메커니즘(샌드박싱, 격리, 모니터링)을 구현할 것
- 시스템을 독립적인 장애 도메인으로 분리하고, 도메인 간 의존성을 최소화할 것
- 로드 밸런싱과 로드 셰딩을 통해 과부하 상황에서 시스템을 보호할 것
- 서킷 브레이커 패턴을 사용하여 연쇄 장애를 방지할 것
- 비핵심 기능에 대한 성능 저하 전략을 미리 정의하고 테스트할 것
- 정기적인 재해 복구 훈련(DiRT)을 실시하여 복원력을 검증할 것
- 소규모 조직은 제어된 성능 저하, 폭발 반경 제어, 장애 도메인 분리에 우선 집중할 것
- 중복성(redundancy)은 신뢰성을 높이지만 복잡성과 비용도 증가시키므로 균형을 맞출 것

## 핵심 인용
> "좋은 시스템 설계는 복원력에 대한 계획을 포함한다: 공격에 대한 방어와 시스템에 스트레스를 주는 비정상적 상황을 견디는 능력이다."

> "방어 심층은 다중의 때로는 중복된 방어 메커니즘의 적용이다. 구별된 장애 도메인은 장애의 '폭발 반경'을 제한하여 신뢰성도 높인다."

> "시스템이 충분히 복잡해지면, 전체 시스템이 침해에 면역이라는 것을 쉽게 증명할 수 없다. 이 문제를 방어 심층과 구별된 장애 도메인으로 부분적으로 해결할 수 있다."
