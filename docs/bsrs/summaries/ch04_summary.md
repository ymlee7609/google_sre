# Design Tradeoffs 요약

## 핵심 원칙
- 보안과 신뢰성은 시스템 설계의 **창발적 속성(emergent property)**이므로 단일 모듈이나 플래그로 구현할 수 없다
- 기능 요구사항과 비기능 요구사항(보안, 신뢰성) 사이의 트레이드오프를 설계 초기에 고려해야 한다
- 보안/신뢰성을 기존 시스템에 사후 적용하면 비용이 크게 증가하고 추가 위험을 초래한다
- **초기 속도(initial velocity)**와 **지속 속도(sustained velocity)**를 구분해야 한다
- 보안/신뢰성 투자를 미루면 초기에는 빠르지만, 후기에는 기술 부채로 크게 느려진다
- 프레임워크 기반 개발은 보안과 신뢰성을 "구조적으로 안전하게(secure by construction)" 만든다
- 보안 위험 완화 시도가 새로운 신뢰성 위험을, 신뢰성 위험 완화가 새로운 보안 위험을 만들 수 있다
- Google의 디자인 문서 템플릿은 확장성, 중복성, 의존성, 데이터 무결성, SLA, 보안/프라이버시를 체계적으로 검토한다

## 주요 프랙티스 및 권고사항
- 결제 처리 같은 민감 데이터는 가능하면 직접 보유하지 말고 제3자 서비스를 활용할 것
- 제3자 의존성 추가 시 새로운 장애 모드와 보안 위험을 함께 평가할 것
- 공통 프레임워크(Google Web Application Framework 등)를 사용하여 보안/신뢰성 관심사를 자동 처리할 것
- 적합성 검사(conformance check)를 통해 코딩 가이드라인 준수를 자동으로 검증할 것
- CI/CD 파이프라인, 테스트 자동화, 점진적 롤아웃 인프라에 초기 투자할 것
- 디자인 문서에 보안과 신뢰성 섹션을 필수로 포함할 것
- 보안 및 SRE 팀을 프로젝트 초기 설계 단계부터 참여시킬 것
- 기능 플래그(feature flag)를 통해 코드와 설정의 분리된 롤아웃을 지원할 것

## 핵심 인용
> "보안과 신뢰성을 고려하지 않고 초기 속도를 최우선으로 선택할 수 있다. 테스트 없이, 타르볼을 프로덕션에 복사하는 릴리스 프로세스로 첫 번째 데모를 빠르게 만들 수 있다. 하지만 세 번째 릴리스쯤에는 일정이 지연되고 기술 부채에 시달릴 것이다."

> "보안을 위한 설계 투자는 일반적으로 큰 선행 투자가 아니라 프레임워크의 제약을 준수하기 위한 점진적이고 보통 적당한 지속적 노력을 수반한다."

> "보안 위험을 완화하려는 시도에서 발생한 신뢰성 위험을 완화하려다 다시 보안 위험이 발생하는 순환적 트레이드오프를 만날 수 있다."
