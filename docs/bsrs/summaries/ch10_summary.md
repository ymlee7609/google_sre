# Mitigating Denial-of-Service Attacks 요약

## 핵심 원칙
- DoS 공격은 서비스의 수요(demand)가 공급(supply, 용량)을 초과하게 만들어 정상 사용자에게 서비스를 제공하지 못하게 한다
- 공격자는 DNS, 네트워크, 프론트엔드, 백엔드 등 의존성 체인의 어느 지점이든 공격할 수 있다
- **증폭 공격(amplification attack)**: 소량의 요청으로 대량의 응답 트래픽을 생성하여 피해자에게 집중시키는 기법
- 계층적 방어(layered defense)가 가장 비용 효율적이며, 외부 계층에서 가능한 한 일찍 공격 트래픽을 차단해야 한다
- 공유 인프라에서의 방어는 규모의 경제를 제공한다
- Anycast를 사용하면 분산 공격을 여러 위치로 분산시켜 단일 데이터센터에 집중되는 것을 방지할 수 있다
- 자체 유발 "공격"(사용자 행동 동기화, 클라이언트 재시도 폭주)도 DoS와 유사한 문제를 일으킬 수 있다
- **Fail static**: 제어 시스템 장애 시 정책을 변경하지 않는 전략이 Fail open/closed보다 안전하다

## 주요 프랙티스 및 권고사항
- 위협 모델을 기반으로 의존성 체인의 최약 링크에 방어를 집중할 것
- 네트워크 엣지에 ACL을 배포하여 의심스러운 트래픽을 조기에 차단할 것
- 캐싱 프록시를 활용하여 정적 콘텐츠 요청이 백엔드에 도달하지 않도록 할 것
- 상태 보유 방화벽(stateful firewall)은 상태 소진 공격에 취약하므로, 라우터 ACL을 우선 사용할 것
- 클라이언트에 지수 백오프(exponential backoff)와 지터(jitter)를 구현할 것
- DoS 완화 시스템 자체도 공격에 대해 복원력이 있어야 한다
- 전략적 대응: 공격자에게 방어 방법을 노출하지 않도록 신중하게 대응할 것
- 제어된 성능 저하(graceful degradation) 모드를 사전에 정의하고 테스트할 것

## 핵심 인용
> "모든 온라인 서비스는 DoS 공격에 대비해야 한다. 각 조직에는 흡수할 수 있는 트래픽의 한계가 있으며, 방어자의 과제는 배포된 용량을 초과하는 공격을 가장 효율적으로 완화하는 것이다."

> "공격 트래픽이 시스템 깊숙이 들어갈수록 더 집중적이고 완화 비용이 더 높아진다. 따라서 각 계층이 뒤의 계층을 보호하는 계층적 방어가 필수적인 설계 특성이다."

> "중앙 제어기가 실패하면, fail closed(모든 트래픽 차단)나 fail open(공격 트래픽 통과)이 아니라 fail static(정책 불변)을 선택한다."
