# Design for Understandability 요약

## 핵심 원칙
- **시스템 이해가능성(understandability)**: 관련 기술 배경을 가진 사람이 시스템의 운영 동작과 보안 속성을 정확하고 자신 있게 추론할 수 있는 정도
- 이해 가능한 시스템은 보안 불변식(security invariant)과 신뢰성 SLO를 달성할 수 있다는 확신을 준다
- **불변식(invariant)**과 **멘탈 모델(mental model)**을 기반으로 시스템을 분석하고 이해해야 한다
- 계층화된 시스템 아키텍처와 표준화된 프레임워크가 이해가능성을 높인다
- 보안 경계(security boundary)는 시스템의 신뢰 영역을 구분하며, 경계 간의 상호작용을 명확히 정의해야 한다
- 복잡성은 이해가능성의 적이며, 시스템이 복잡할수록 보안/신뢰성 추론이 어려워진다
- API 설계와 애플리케이션 프레임워크는 보안/신뢰성 속성에 대한 추론 능력에 큰 영향을 미친다
- "구조적으로 안전한(secure by construction)" 설계가 "올바르게 사용하면 안전한(correct usage)" 설계보다 우월하다

## 주요 프랙티스 및 권고사항
- 시스템의 보안 불변식을 명시적으로 문서화하고, 위반 여부를 자동으로 검증할 것
- 신뢰 경계를 명확히 정의하고, 경계를 넘는 모든 데이터를 검증할 것
- XSS, SQL 인젝션 등 취약점을 방지하는 타입 안전(type-safe) API를 사용할 것
- 프레임워크 수준에서 보안 속성을 보장하여 개별 개발자가 실수할 여지를 줄일 것
- 시스템을 독립적으로 추론할 수 있는 작은 구성 요소로 분해할 것
- Identity-Aware Proxy(IAP)처럼 접근 제어를 중앙 집중화할 것
- 코드 리뷰에서 보안 불변식 위반을 탐지할 수 있는 정적 분석 도구를 활용할 것
- 멘탈 모델을 단순하게 유지하고, 예외 사항을 최소화할 것

## 핵심 인용
> "시스템의 이해가능성은 보안과 신뢰성의 기반이다. 시스템을 이해할 수 없다면, 그것이 안전하거나 신뢰할 수 있는지 확신할 수 없다."

> "'구조적으로 안전한' 접근 방식은 API나 프레임워크의 설계 자체가 특정 유형의 취약점을 불가능하게 만든다. 이는 개발자에게 올바른 사용법을 교육하는 것보다 훨씬 효과적이다."

> "복잡성은 보안의 적이다. 시스템이 복잡할수록 보안 속성을 추론하기 어려워지고, 예상치 못한 상호작용으로 인한 취약점이 발생할 가능성이 높아진다."

## 관련 키워드
이해가능성, 보안 불변식, 멘탈 모델, 신뢰 경계, 구조적 안전성, 타입 안전 API, XSS 방지, Identity-Aware Proxy, 계층 아키텍처, 프레임워크, 정적 분석, 복잡성, 공격 표면 최소화, SafeHTML
