# Case Study: Designing, Implementing, and Maintaining a Publicly Trusted CA 요약

## 핵심 원칙
- 공개 신뢰 인증 기관(CA)은 인터넷 전송 계층의 신뢰 앵커 역할을 하며, TLS 인증서를 발급한다
- Google은 제3자 의존성, 자동화 필요성, 비용 효율성 등의 이유로 자체 공개 신뢰 CA를 구축했다
- 3계층 아키텍처(요청 파싱, 등록 기관 기능, 인증서 서명)로 역할을 분리했다
- 이중 신뢰 영역(dual trust zone): 신뢰할 수 없는 입력은 민감한 작업과 다른 환경에서 처리한다
- 단순성이 핵심 설계 원칙이며, 표준의 전체 기능이 아닌 필요한 기능만 구현했다
- 프로그래밍 언어 선택이 보안에 직접적 영향을 미침 (Go의 메모리 안전성 활용)
- CA 키 자료의 도난이나 오용이 가장 심각한 위험이다
- 지속적 검증(continuous validation)과 Certificate Transparency를 통해 오발급을 방지한다

## 주요 프랙티스 및 권고사항
- 신뢰할 수 없는 입력을 처리하는 부분에는 메모리 안전 언어(Go 등)를 사용할 것
- 마이크로서비스로 분해하되, 과도한 분해로 인한 유지보수 비용 증가를 경계할 것
- 제3자 및 오픈소스 코드에 대해 심층 보안 리뷰를 수행하고, 발견된 문제에 대한 패치를 기여할 것
- 폐쇄 소스 벤더 코드는 nsjail 등으로 격리하여 실행할 것
- 단위 테스트, 통합 테스트 외에 퍼징(fuzzing)과 정적 분석을 적용할 것
- 긍정적/부정적 테스트 조건을 모두 포함하는 테스트 저장소를 구축할 것
- 루트 키 자료는 오프라인으로 유지하고, 다중 물리적 보호 계층과 이중 인가를 적용할 것
- 인증서 발급 전 여러 단계에서 린터(linter)를 통한 자동 검증을 수행할 것

## 핵심 인용
> "오픈소스 패키지가 보안 맥락에서 널리 사용되고 강력한 보안 배경을 가진 개인이나 조직에서 유래하더라도 취약점에 노출될 수 있다."

> "우리는 상용 및 오픈소스 CA 소프트웨어가 우리에게 필요 없는 난해한 속성과 확장을 수용하려는 시도가 시스템의 복잡성을 증가시켜 검증을 어렵게 하고 오류에 취약하게 만든다는 것을 발견했다."

> "좋은 원칙을 처음부터 확립하는 것이 중요하다. 작게 시작하고, 시스템의 보안과 신뢰성 속성을 지속적으로 반복 개선하여 장기적 지속가능성을 달성할 수 있다."
